给定一个问题类别和准确率目标，搭建一套 debate 协议。

产出：

1. 使用不同提示词（最好也使用不同模型）的 `Debater`，以避免同质化。
2. 轮次执行器：全连接网状、星型或环型拓扑。
3. 收敛规则：多数投票、按置信度加权，或带回退的绝对多数。
4. 第 1 轮强制分歧：在可能的情况下，每名辩手都返回一个互不相同的提案。
5. 成本核算：每个问题的批判操作总数 + token 成本。

硬性拒绝项：

- 所有辩手都使用相同提示词且相同模型。必然导致群体思维。
- N >= 6 时不核查成本就采用全连接网状。Debate 操作以 O(N*R) 增长。
- 没有收敛规则。返回辩手 0 在第 R 轮的答案并不算收敛。

拒绝规则：

- 如果产品对延迟敏感（预算 <1s），拒绝 debate。改用 Self-Refine（第 05 课）或并行投票（第 12 课）。
- 如果问题类别是简单的事实查询（首都、日期、定义），拒绝 debate。查询 + CRITIC（第 05 课）更便宜。
- 如果在评估集的任意问题上，辩手们在第 1 轮后都没有分歧，拒绝该协议。你需要模型/提示词的多样性。

输出：`debater.py`、`topology.py`、`convergence.py`、`runner.py`、`README.md`，其中说明 N/R 的选择、拓扑选型理由，以及在评估集上的成本-准确率权衡度量。结尾给出“接下来读什么”：若任务更简单，指向第 12 课（工作流模式）；若要将 debate 嵌入更大的系统，则指向第 28 课（编排模式）。
