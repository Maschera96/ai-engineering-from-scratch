给定一个用户请求和一个工具目录（名称、输入模式、描述），生成一个 ReWOO 计划：由若干步骤组成的 DAG，其中包含工具调用和证据引用（`#E1`、`#E2`、……）。在将计划交给执行器之前先进行验证。

生成：

1. 一个计划 DAG。每个节点都有 id（`E1`、`E2`、……）、工具名称、参数字典（字符串中可包含 `#E<k>` 引用），以及可选的 `parallel_group` 标签。
2. 验证输出。通过拓扑排序进行无环检查；引用解析检查（每个 `#E<k>` 都有一个在前的生产者）；工具存在性检查（每个工具名称都在目录中）；参数模式检查（每个参数都匹配该工具的输入模式）。
3. 并行性提示。对于每个拓扑层级，列出可以并发执行的节点。
4. Planner/solver 拆分建议。如果计划少于 3 步，则建议改用 ReAct。如果计划存在无界循环需求（每一步都需要重新规划），则建议使用带 replanner 的 Plan-and-Execute。如果计划超过 30 步，或面向 web/移动端，则建议使用配合合成计划数据的 Plan-and-Act。

硬性拒绝：

- 含有环的计划。ReWOO 假设是 DAG；环属于 ReAct 或 LATS 的范畴。
- 引用了在拓扑顺序中尚不存在的 `#E<k>`（`k` 尚不存在）的计划。需给出导致失败的具体边。
- 调用了目录中不存在的工具的计划。不要为了让计划成立而臆造工具。
- 引用的参数类型与工具模式不匹配的计划（例如 `#E1` 代入的是字符串，但工具期望的是 int）。

拒绝规则：

- 如果任务是开放式探索（所需工具未知、步骤未知），则拒绝，并建议改用 ReAct 或 LATS（第 04 课）。
- 如果工具目录中包含破坏性工具但没有把关审批工具，则拒绝，并指向第 09 课（权限、沙箱）。

输出：一份结构化计划（JSON 或 YAML）、一份验证报告、一张并行性映射，以及一个后续动作——指向执行器（ReWOO Worker）、replanner（Plan-and-Execute）或更大的轨迹采样循环（Plan-and-Act）。

最后附上一条“接下来读什么”的提示：如果该任务类型此前已被尝试过，则指向第 03 课（Reflexion）；如果计划能从搜索中受益，则指向第 04 课（LATS）。
