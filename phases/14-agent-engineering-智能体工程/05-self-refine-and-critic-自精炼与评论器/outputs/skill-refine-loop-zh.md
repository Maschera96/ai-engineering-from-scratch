给定一个任务、一个迭代预算，以及可用的验证器类型（工具支撑的或仅自评估），输出用于评估器-优化器循环的提示词和停止策略。

产出：

1. 生成器提示词。用于首次输出的确定性生产者。明确说明任务、输出格式和约束。
2. 评估器/验证器提示词。如果有工具可用（搜索、代码运行、测试、计算器、类型检查），说明如何调用它们以及如何产出结构化的评论（JSON，包含：pass/fail、violations[]、suggested_fixes[]）。如果只有自评估可用，明确标注 Self-Refine 的橡皮图章（rubber-stamp）风险，并使用结构上不同的提示词风格（例如对抗式的“至少找出一个缺陷”）。
3. 精炼器提示词。必须引用先前的输出和评论（历史）。明确说明“不要重复先前迭代中标注过的失败模式”是强制要求。
4. 停止策略。以下条件的合取：验证器通过 OR（自评估认为没问题 AND 迭代次数 >= 2）OR 迭代次数 >= max_iterations。绝不使用单一条件。
5. 可观测性钩子。按照第 23 课，将每次迭代记录为一个 OpenTelemetry GenAI span（evaluate、optimize），使完整的精炼轨迹可审计。

硬性拒绝：

- 生成器和评论器使用相同的提示词。橡皮图章风险——模型会认同自己。
- 没有迭代上限。无限的精炼循环会烧掉 token；默认始终上限为 4。
- 要求自由文本散文反馈的验证器提示词。只能是结构化 JSON——pass/fail 加上逐项列出的 violations。
- 从精炼器提示词中丢弃历史。论文表明，没有历史时质量会崩溃。

拒绝规则：

- 如果任务没有验证器，也没有办法构建一个，则拒绝 CRITIC，并指出 Self-Refine 是可用的较弱选项——警告用户存在橡皮图章风险。
- 如果 max_iterations >= 10，则拒绝并建议重新设计任务架构。超过 3-4 趟的精炼到收敛通常是生成器提示词有问题的信号。
- 如果验证器会调用破坏性工具（shell、git 写入），则拒绝并要求设置沙箱边界（第 09 课）。

输出：一个单独的配置块，包含所有提示词、停止策略和工具列表，外加一条“接下来读什么”的提示，根据部署目标指向第 16 课（OpenAI Agents SDK guardrails）、第 12 课（Anthropic 评估器-优化器）或第 30 课（eval 驱动的智能体开发）。
