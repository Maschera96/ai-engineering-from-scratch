给定一个任务类别（预期时长、步骤数量、触发类型、延迟预算），选择运行时形态。

决策：

1. < 30s，用户等待 -> **请求-响应（request-response）**。
2. 渐进式 UX 或语音 -> **流式（streaming）**。
3. 数分钟到数小时，用户不等待 -> **基于队列（queue-based）**。
4. 对外部事件做出反应 -> **事件驱动（event-driven）**。
5. 周期性维护 -> **定时（cron）**。
6. 上述任意一种且重启成本高 -> 增加**持久化执行（durable execution）**。

产出：

1. 在你的技术栈中搭建该形态的脚手架。
2. 可观测性：OTel GenAI spans（第 23 课），接入后端（第 24 课）。
3. 对于队列：DLQ + 重试策略 + 队列深度指标。
4. 对于事件：显式的订阅者注册表 + 回放路径。
5. 对于定时：锁文件或分布式锁，以防止运行重叠。
6. 对于持久化：checkpointer 后端 + 恢复语义。

硬性拒绝：

- 用同步 HTTP 处理一个 5 分钟的任务。用户会挂断；worker 会堆积。
- 基于队列但没有 DLQ。失败的任务会消失。
- 后台工作没有 trace 导出。在用户抱怨之前，故障都不可见。
- “没有持久化状态，我们重试就好。”长周期任务必须做检查点。

拒绝规则：

- 如果产品有 SLA + 回放需求，拒绝 swarm 拓扑 + 非持久化运行时。
- 如果任务受合规约束，拒绝没有审计轨迹的事件驱动方案。
- 如果用户想要定时任务且不加锁，拒绝。运行重叠的定时任务往好了说是重复工作，往坏了说是数据损坏。

输出：运行时脚手架 + 可观测性钩子 + 包含 SLA、重试策略、checkpointer 选型的 README。结尾以“接下来读什么”指向第 23 课（OTel）、第 24 课（可观测性）或第 17 课（用于托管式长时运行的 Managed Agents）。
