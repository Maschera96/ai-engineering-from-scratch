# 混合记忆：向量 + 图 + KV（Mem0）

> Mem0（Chhikaret al.，2025）将记忆视为三个并行的存储——向量用于语义相似度，KV 用于快速事实查找，图用于实体关系推理。检索时由一个评分层融合三者。这是 2026 年外部记忆的生产标准。

**类型：** 构建
**语言：** Python（标准库）
**前置知识：** Phase 14 · 07（MemGPT）、Phase 14 · 08（LettBlocks）
**时长：** 约 75 分钟

## 学习目标

- 解释为什么单一存储（仅向量、仅图、仅 KV）不足以支撑智能体记忆。
- 说出 Mem0 的三个并行存储，以及每个分别针对什么做了优化。
- 描述 Mem0 的融合评分——相关性、重要性、时近性——以及为什么它是加权求和，而不是层级关系。
- 用标准库实现一个玩具版三存储记忆，其 `add()` 会写入全部三个存储，而 `search()` 会融合各方结果。

## 问题

对于三类查询中的某一类，单一存储总是错的：

- **语义相似度**——“我们上周关于智能体漂移讨论了什么？”向量获胜；KV 和图都查不到。
- **事实查找**——“用户的电话号码是多少？”KV 获胜；向量浪费，图则过度设计。
- **关系推理**——“哪些客户共用同一个计费实体？”图获胜；向量和 KV 都答不上来。

生产级智能体在一个会话里会同时发起这三类查询。单一存储的记忆对其中两类总是错的。Mem0 的贡献在于把三者接到统一的 `add`/`search` 接口背后，并用一个评分函数把它们融合起来。

## 概念

### 三个并行存储

Mem0（arXiv:2504.19413，2025 年 4 月）在 `add(text，user_id，元数据)` 上：

1。从文本中抽取候选事实（一个由 LLM 驱动的步骤）。
2。把每个事实写入向量存储（嵌入）以支持语义搜索。
3。把每个事实以 (user_id，fact_type，entity) 为键写入 KV 存储，实现 O(1) 查找。
4。把每个事实作为带类型的边写入图存储（Mem0g），以支持关系查询。

在 `search(query，user_id)` 上：

1。向量存储按嵌入余弦相似度返回 top-k。
2。KV 存储按查询推导出的 (user_id，type，entity) 返回直接命中。
3。图存储返回从查询实体可达的子图。
4。一个评分层融合三者。

### 融合评分

```
score = w_relevance * relevance(q，记录)
      + w_importance * importance(记录)
      + w_recency * recency(记录)
```

- **相关性**——向量余弦相似度、KV 精确匹配、图路径权重。
- **重要性**——写入时打标或学习得到（有些事实更重要：姓名、ID、政策）。
- **时近性**——自上次写入或读取以来随时间的指数衰减。

权重按产品调优。聊天智能体的 `w_recency` 更高；合规智能体的 `w_importance` 更高；检索智能体的 `w_relevance` 更高。

### Mem0g 与时序推理

Mem0g 增加了一个冲突检测器。当新事实与已有的边矛盾时，已有的边会被标记为无效，但不会被删除。时序查询（“用户三月份在哪个城市？”）会遍历“在该时间有效”的子图。

这正是 Lett的失效模式所泛化的那种合规级行为。

### 基准数字

Mem0 论文报告（2025）：

- **LoCoMo**（长篇对话记忆）：91.6
- **LongMemEval**（长跨度情景记忆）：93.4
- **BEAM 1M**（百万 token 记忆基准）：64.1

对比基线（128k 全上下文 LLM、扁平向量存储、扁平 KV）都落后 10 分以上。基准本身不足以决定选型——运营形态才是关键——但这些数字表明，融合设计带来的差异不是误差范围内的小波动。

### 作用域分类

Mem0 按作用域划分记忆：

- **用户记忆**——跨会话持久化，以 `user_id` 为键。
- **会话记忆**——在单个线程内持久化。
- **智能体记忆**——每个智能体实例的状态。

每次写入都要选定一个作用域。检索可以跨作用域查询，并为各作用域设置不同权重。不加思考地混用作用域，正是“助手把 Bob 的项目告诉了 Alice”这类事故的根源。

### 这个模式会在哪里出错

- **嵌入漂移。** 向量结果在头一百次查询里看着没问题，但随着语料增长会逐渐退化。为使用最多的 top-N 记录加上周期性重新嵌入。
- **KV 模式蔓延。** `(user_id，type，entity)` 看似简单，直到每个团队都加上自己的 `type`。每季度审计一次 type 集合。
- **图爆炸。** 一个嘈杂的抽取器每条消息加 50 条边。给每次 `add` 调用的图写入设上限；丢弃低置信度的边。

## 动手构建

`code/main.py` 用标准库实现了三存储模式：

- `VectorStore`——用朴素的 token 重叠相似度作为嵌入的替身。
- `KVStore`——以 `(user_id，fact_type，entity)` 为键的字典。
- `GraphStore`——带类型的边（subject，relation，object，valid）。
- `Mem0`——顶层门面，提供 `add()`、`search()`、融合评分以及作用域感知的检索。
- 一段在多用户、多会话对话上的完整运行追踪。

运行它：

```
python3 code/main.py
```

输出展示了三条独立的召回路径，外加融合后的 top-k。修改 `main()` 顶部的评分权重，观察排序如何变化。

## 投入使用

- **Mem0（Apache 2.0）**——生产就绪。用 Postgres + Qdrant + Neo4j 自托管，或使用托管云。
- **Letta**——三层 core/recall/archival；自带向量和图后端。
- **Zep**——商业替代方案，带时序知识图谱和事实抽取。
- **自建**——当你需要对抽取器（合规）或融合权重（时近性占主导的语音智能体）做精确控制时。

## 练习

1。用真实的嵌入模型（sentence-transformers、Ollama、OpenAI embeddings）替换玩具版向量相似度。在一段合成的长对话上测量 recall@10。排序会在 1000 次写入后漂移吗？
2。增加一个时序查询：`search(query，as_of=timestamp)`。只返回在该时间或之前有效的记录。哪个存储要改动最多？
3。实现一个冲突检测器：如果传入的事实与某条图边矛盾，使旧边失效并把两者都记录下来。在“用户住在柏林”->“用户住在里斯本”上测试。
4。把融合评分器移植到包含一个 `user_feedback` 维度（对检索到的记录点赞）。你如何防止被钻空子（智能体只返回它已经喜欢的记录）？
5。阅读 Mem0 文档（`docs.mem0.ai`）。把玩具版移植到 `mem0` 客户端调用上。在同样的 20 个测试查询上对比检索质量。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| 混合记忆 | “向量加图加 KV” | 三个存储并行写入，检索时融合 |
| 事实抽取 | “记忆摄取” | 把文本拆成 (entity，relation，fact) 元组的 LLM 步骤 |
| 融合评分 | “相关性排序” | 相关性、重要性、时近性的加权求和 |
| 作用域 | “记忆命名空间” | user / session / agent——决定谁能看到什么 |
| Mem0g | “记忆图” | 带时序有效性的带类型边，用于关系查询 |
| 时序失效 | “软删除” | 把矛盾的边标记为无效；永不删除 |
| 嵌入漂移 | “检索腐化” | 向量质量随语料增长而退化；定期重新嵌入 |
