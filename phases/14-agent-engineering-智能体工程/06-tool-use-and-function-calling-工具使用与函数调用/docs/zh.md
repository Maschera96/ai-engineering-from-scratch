# 工具使用与函数调用

> Toolformer（Schick 等，2023）开创了自监督的工具标注。Berkeley Function Calling Leaderboard V4（Patil 等，2025）确立了 2026 年的标杆：40% agentic、30% 多轮、10% live、10% non-live、10% 幻觉。单轮调用已被解决。记忆、动态决策和长程工具链尚未解决。

**类型：** 构建
**语言：** Python (stdlib)
**前置知识：** Phase 14 · 01（智能体循环）、Phase 13 · 01（函数调用深入）
**时长：** 约 60 分钟

## 学习目标

- 解释 Toolformer 的自监督训练信号：仅当执行能降低下一个 token 的损失时才保留工具标注。
- 说出 BFCL V4 的五个评估类别以及各自衡量什么。
- 实现一个仅用 stdlib 的工具注册表，包含模式验证、参数强制转换和执行沙箱。
- 诊断 2026 年的三大开放问题：长程工具链、动态决策和记忆。

## 问题

早期的工具使用问的是：模型能否预测出一个正确的函数调用？现代的工具使用问的是：模型能否在 40 步内串联工具，带着记忆、在部分可观察性下、能从工具失败中恢复，且不臆造出根本不存在的工具？

Toolformer 确立了基线：模型可以通过自监督学习何时调用工具。BFCL V4 定义了 2026 年的评估目标。两者之间的差距正是生产级智能体所处的空间。

## 概念

### Toolformer（Schick 等，NeurIPS 2023）

思路：让模型用候选 API 调用来标注自己的预训练语料。对每个候选，执行它。仅当包含工具结果能降低下一个 token 的损失时才保留该标注。然后在过滤后的语料上微调。

涵盖的工具：计算器、QA 系统、搜索引擎、翻译器、日历。自监督信号纯粹关乎工具是否有助于预测文本——没有人工标注。

规模化结果：工具使用在规模化时涌现。较小的模型会因工具标注受损；较大的模型则获益。这就是为什么 2026 年的前沿模型内置了强大的工具使用能力，而大多数 7B 模型需要显式的工具使用微调才能可靠。

### Berkeley Function Calling Leaderboard V4（Patil 等，ICML 2025）

BFCL是2026 年事实上的评估标准。V4 构成：

- **Agentic（40%）**——完整的智能体轨迹：记忆、多轮、动态决策。
- **多轮（30%）**——带工具链的交互式对话。
- **Live（10%）**——用户提交的真实提示（更难的分布）。
- **Non-Live（10%）**——合成测试用例。
- **幻觉（10%）**——检测何时不应调用任何工具。

V3 引入了基于状态的评估：在一段工具序列之后，检查 API 的实际状态（例如"文件是否已创建？"），而不是匹配工具调用的 AST。V4 增加了 web 搜索、记忆和格式敏感性类别。

2026 年的关键发现：单轮函数调用已接近被解决。失败集中在记忆（跨轮携带上下文）、动态决策（根据先前结果选择工具）、长程工具链（20+ 步后漂移）以及幻觉检测（在没有合适工具时拒绝调用）。

### 工具模式

每个提供商都有自己的模式。它们在细节上有差异，但共享相同的形态：

```
name：string
description：string (what它does，when to use it)
input_schema：JSON Schem(properties，required，types，enums)
```

Anthropic 直接使用 `input_schema`。OpenAI 使用 `function.parameters`。两者都接受 JSON Schema。描述是承重的——模型读取它们来挑选正确的工具。糟糕的工具描述是"选错工具"类失败的头号根源。

### 参数验证

不要信任任何工具调用。要验证：

1。**类型强制转换。** 模式说是 int，但模型可能返回字符串 "5"。如果无歧义就强制转换；否则拒绝。
2。**枚举验证。** 如果模式说 `status in {"open"，"closed"}`，而模型发出 `"in_progress"`，则用描述性错误拒绝。
3。**必填字段。** 缺失必填字段 -> 立即把错误观察返回给模型，而不是崩溃。
4。**格式验证。** 日期、邮箱、URL——用具体的解析器验证，而非正则。

每一次验证失败都应返回一个结构化观察，让模型能以正确的形态重试。

### 并行工具调用

现代提供商支持在一个助手轮次中并行调用工具。该循环为：

1。模型发出 3 个工具调用，各带不同的 `tool_use_id`。
2。运行时执行它们（若相互独立则并行）。
3。每个结果作为一个 `tool_result` 块返回，并通过 `tool_use_id` 关联。

工程规则：把关联 ID 当作承重组件。把它们调换，你就会得到"错误工具对错误结果"的路由。

### 沙箱

工具执行是沙箱边界。详见第 09 课。简短版本：每个工具都应指定读/写面、网络访问、超时、内存上限。通用的 `run_shell(cmd)` 是危险信号；具体的 `git_status()` 更安全。


## 实际使用

每个提供商都有自己的工具模式——Anthropic、OpenAI、Gemini、Bedrock。如果你需要多提供商，请使用翻译层（OpenAI Agents SDK、Vercel AI SDK、LangChain 工具适配器）。BFCL 是参考基准——如果工具使用是产品的核心，发布前请用它来测试你的智能体。

## 练习

1。添加一个"空操作"工具，让模型可以显式拒绝使用任何其他工具。在一个类 BFCL 的幻觉测试上衡量它。
2。为 int-as-string 和 float-as-string 实现参数强制转换。强制转换从哪里开始会掩盖真正的 bug？
3。添加每个工具的超时和断路器（连续失败 3 次后拒绝该工具 60 秒）。这对模型的恢复方式有何改变？
4。阅读 BFCL V4 描述。挑一个类别（例如"多轮"）并通过你的智能体跑 10 个示例提示。报告通过率。
5。把 stdlib 验证器移植到 Pydantic 或 Zod。Pydantic/Zod 抓到了哪些玩具版漏掉的东西？

## 关键术语

| 术语 | 人们怎么说 | 它实际的含义 |
|------|----------------|------------------------|
| Function calling | "工具使用" | 带验证模式的结构化输出工具调用 |
| Toolformer | "自监督工具标注" | Schick 2023——保留那些结果能降低下一个 token 损失的工具调用 |
| BFCL | "Berkeley Function Calling Leaderboard" | 2026 基准：40% agentic、30% 多轮、10% live、10% non-live、10% 幻觉 |
| 工具模式 | "给模型的函数签名" | 名称、描述、参数的 JSON Schem|
| tool_use_id | "关联 ID" | 把一个工具调用与其结果绑定；对并行分发至关重要 |
| 幻觉检测 | "知道何时不调用" | V4 类别：在没有合适工具时拒绝调用 |
| 参数强制转换 | "字符串转 int 修复" | 针对可预测的模式不匹配的窄修复；有歧义则拒绝 |
| 沙箱 | "工具执行边界" | 每个工具的读/写面、网络、超时、内存上限 |
