给定一个仓库、智能体产品及其依赖面，生成一个项目专属的初始化脚本以及 CI 接线。

产出：

1. `tools/init_agent.py`，包含这些探测：运行时版本、列出的依赖、测试命令可解析性、必需的环境变量、状态文件新鲜度。
2. 在脚本旁记录 `init_report.json` 的模式。每项探测返回 `(name, status: pass|warn|fail, detail)`。
3. `.github/workflows/agent-init.yml`（或等价文件），运行该脚本，并在任何 fail 级别的探测上阻断智能体作业。
4. 一个 `pre-task` 钩子脚本，供智能体运行时在每个会话启动前调用。
5. 在 `docs/init.md` 中编写文档，列出每项探测、其严重级别，以及如何修复某个失败。

硬性拒绝：

- 在没有超时的情况下向网络发起调用的探测。初始化必须快速且离线安全。
- 需要 LLM 调用的探测。初始化是确定性的管道工作。
- 被封装器吞掉的非零退出码。大声失败正是整件事的关键所在。
- 在不具备幂等性的情况下触碰状态的探测。连续两次运行必须产出除时间戳外完全相同的报告。

拒绝规则：

- 如果项目没有测试命令，拒绝交付该脚本。改为将此缺口加入 workbench 审计。
- 如果环境变量列表包含脚本将要打印的密钥，拒绝并强制脱敏。初始化报告绝不应携带密钥。
- 如果某项探测在试运行中耗时超过三秒，在交付前披露该耗时发现。冗长的探测会把初始化变成形式主义。

输出结构：

```
<repo>/
├── tools/
│   ├── init_agent.py
│   └── pre_task.sh
├── docs/
│   └── init.md
└── .github/
    └── workflows/
        └── agent-init.yml
```
