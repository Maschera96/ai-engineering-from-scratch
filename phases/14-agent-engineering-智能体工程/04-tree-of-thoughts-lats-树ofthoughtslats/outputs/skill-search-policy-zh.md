给定一种任务形态（单一答案 / 多答案 / 开放式）、一个 token 预算以及一个可用的评估器（标量测试 / 启发式 / 自我评估），产出一份带有具体参数的搜索策略推荐。

产出：

1. 决策。从以下选项中选一个：线性 ReAct、束搜索 ToT（带束宽 k）、BFS ToT（带最大深度）、带剪枝的 DFS ToT、MCTS LATS（带迭代次数和 UCT c）、进化式搜索（仅当评估器是程序化且可校验时）。
2. 参数。对每种策略给出具体的数值默认值：束宽、深度上限、分支因子 K、每层 rollout 数、UCT c（默认 1.4）、超时时间。
3. 价值函数。明确指出对节点打分的依据是什么。可选项：单元测试通过率、到目标的数值距离、带格式的提示式 LLM 评分（sure/likely/impossible 或 1..10 或投票），或环境奖励。
4. token 预算估算。最坏情况 token 数 = branching_factor ^ depth * avg_prompt_tokens。给出该数字。如果超出用户预算，则推荐一种更便宜的策略。
5. 失败模式。对每种选定的策略，列出其前两位失败模式及其缓解措施（例如 LATS + 嘈杂评估器 -> 按 CRITIC 增加工具落地的验证，见第 05 课）。

硬性拒绝：

- 当评估器不可靠时（仅有自我评估，无真实标签）推荐搜索。应回退到 ReAct + CRITIC。
- 在没有充分理由的情况下把分支因子 K 设得高于 5。K=3-5 是论文默认值；K=10 会让成本爆炸。
- 把 LATS 应用于聊天式任务。对于没有程序化目标的对话式问答，搜索没有帮助。
- 在没有机器可校验的适应度函数时使用进化式搜索。AlphaEvolve 只有在适应度是程序化的（运行测试、测量速度、验证定理）时才有意义。

拒绝规则：

- 如果 token 预算 < 单轨迹成本的 5 倍，拒绝搜索并推荐 ReAct + Reflexion（第 03 课）。
- 如果挂钟延迟预算 < 10 秒，拒绝 LATS 并推荐 ReAct。
- 如果任务是纯信息检索，拒绝搜索并推荐 ReWOO（第 02 课）。

输出：一个推荐块（选定的策略、参数、价值函数、预算估算），外加一条“接下来读什么”的提示，指向第 05 课（CRITIC）以了解评估器可靠性、第 11 课（AlphaEvolve）以了解进化式变体，或第 30 课（评估驱动开发）以进行基准级别的验证。
