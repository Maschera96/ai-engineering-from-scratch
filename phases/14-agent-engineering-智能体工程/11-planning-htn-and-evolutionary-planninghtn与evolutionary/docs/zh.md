# 使用 HTN 与进化式搜索进行规划

> 符号化规划处理那些可证明正确的场景。进化式代码搜索处理那些适应度函数可由机器校验的场景。ChatHTN（2025）与 AlphaEvolve（2025）展示了它们各自与 LLM 配对时所能释放的能力。

**类型：** 构建
**语言：** Python（标准库）
**前置知识：** Phase 14 · 02（ReWOO 与 Plan-and-Execute）
**时长：** 约 75 分钟

## 学习目标

- 解释层次任务网络（Hierarchical Task Networks）：任务、方法、操作符、前置条件、效果。
- 描述 ChatHTN 的混合循环——符号化搜索配合 LLM 回退式分解。
- 解释 AlphaEvolve 的进化循环，以及为何它只在配备程序化评估器时才有效。
- 用标准库实现一个玩具级 HTN 规划器外加一个玩具级进化式搜索。

## 问题所在

ReWOO（第 02 课）、Plan-and-Execute 和 ReAct 覆盖了大多数智能体规划。有两类场景它们处理得并不好：

1。**具备可证明正确性的计划。** 排程、航线规划、合规工作流——计划必须在构造上即是可靠的。一个流畅但偶尔会幻觉出某一步骤的 LLM 计划是不可接受的。
2。**带有机器可校验适应度函数的优化。** 矩阵乘法、调度启发式、编译器优化遍——目标不是"一个正确的计划"，而是"最优的计划"。

HTN 规划与 AlphaEvolve 解决的是这两个不同的问题。两者都把 LLM 当作放大器，而非替代品。

## 概念

### 层次任务网络

一个 HTN 包含：

- **任务（Tasks）**——复合任务（需要被分解）与原子任务（可直接执行）。
- **方法（Methods）**——将复合任务分解为子任务的方式，附带前置条件。
- **操作符（Operators）**——带有前置条件和效果的原子动作。
- **状态（State）**——一组事实。

规划：给定一个目标任务和一个初始状态，找到一个分解，使其分解为原子操作符序列，且每个操作符的前置条件按顺序得到满足。

HTN 比 LLM 更古老，至今仍是可证明正确计划的参照标准。

### ChatHTN（Gopalakrishnan 等人，2025）

ChatHTN（arXiv:2505.11814）将符号化 HTN 与 LLM 查询交错进行：

1。尝试用现有方法分解当前的复合任务。
2。如果没有适用的方法，则询问 LLM："在状态 `s` 下你会如何分解 `task`？"
3。将 LLM 的响应翻译成候选子任务。
4。对照操作符模式进行校验；拒绝无效的分解。
5。递归。

论文的核心论断：所产生的每个计划都是可证明可靠的，因为 LLM 的建议仅作为候选分解进入流程，绝不会作为对计划的直接编辑。符号化层负责正确性；LLM 则扩展方法库。

在线方法学习（OpenReview `gwYEDY9j2x`，2025 年的后续工作）增加了一个学习器，通过回归对 LLM 生成的分解进行泛化——将 LLM 查询频率削减最多达 75%。

### AlphaEvolve（Novikov 等人，2025）

AlphaEvolve（arXiv:2506.13131，DeepMind，2025 年 6 月）是另一种性质截然不同的方法：由 Gemini 2.0 Flash/Pro 集成编排的进化式代码搜索。

循环：

1。从一个种子程序 + 一个程序化评估器（返回适应度分数）开始。
2。LLM 集成提出突变。
3。让突变经过评估器运行。
4。保留最优者；再次突变。

已发表的成果：

- 56 年来首次超越 Strassen 算法实现 4x4 复数矩阵乘法（48 次标量乘法）。
- 通过一个 Borg 调度启发式为 Google 收回了 0.7% 的算力。
- 在前沿工作负载上实现 FlashAttention 提速 32%。

硬性约束：适应度函数必须是机器可校验的。在散文式答案上做进化式搜索是不会收敛的。

### 何时使用哪一种

| 问题类别 | 使用 | 原因 |
|---------------|-----|-----|
| 带硬约束的排程 | HTN + ChatHTN | 可证明的可靠性 |
| 编译器优化 | AlphaEvolve | 机器可校验的适应度 |
| 多步骤任务执行 | ReAct / ReWOO | LLM 在循环中，无形式化保证 |
| 带测试的代码改进 | AlphaEvolve | 测试即评估器 |
| 受策略约束的自动化 | HTN | 前置条件编码了策略 |

### 这种模式会在哪里出错

- **没有操作符的 HTN。** 没有前置条件/效果模式，可靠性论断就会崩塌。ChatHTN 的"LLM 建议分解"要求模式能够拒绝无效的动作。
- **没有真正评估器的 AlphaEvolve。** "问 LLM 这段代码是否更好"并不是适应度函数。评估器必须是确定性且快速的。
- **过度工程化。** 大多数智能体任务两者都不需要。先考虑 ReAct 或 ReWOO。

## 动手构建

`code/main.py` 实现了两个玩具级示例：

- 一个标准库 HTN 规划器，带有操作符、方法、前置条件、效果，以及一个 `LLMFallback`，当没有方法匹配某个复合任务时它会介入。这个"LLM"是一个脚本化的分解器，使规划器能离线运行。
- 一个对算术程序进行进化式搜索的标准库实现：生长表达式，使其输出在测试集上最小化 `|f(x) - target|`。评估器是确定性的。

运行它：

```
python3 code/main.py
```

该轨迹展示了 HTN 规划器分解一个复合任务（中途带有一次 LLM 回退），以及进化循环收敛到一个目标表达式。

## 实际使用

- **HTN 规划器**——`pyhop`、`SHOP3`，或为领域特定的策略执行自行构建。
- **ChatHTN**——研究代码；其模式（符号化 + LLM 回退）可干净地移植到任何 HTN 规划器。
- **AlphaEvolve**——DeepMind 论文；其模式（集成 + 评估器）可复现。OpenEvolve 等类似的开源分支正在涌现。
- **智能体框架**——目前还没有任何框架内置一流的 HTN 或 AlphaEvolve 支持。可将其构建为子智能体或后台工作进程。

## 交付

`outputs/skill-hybrid-planner.md` 生成一个混合规划器脚手架（HTN 或进化式），并对 LLM 的角色作明确界定。

## 练习

1。为 HTN 规划器扩展回溯能力：当某个操作符的后置条件在运行时失败时，回滚并尝试下一个方法。
2。为 ChatHTN 添加一个 LLM 方法缓存：当 LLM 在状态模式 `P` 下分解任务 `T` 时，存储结果。下次调用时先重新检查方法库。
3。将进化式搜索的评估器换成一个真实的测试套件。进化出一个能通过 20 个测试用例的排序函数；报告收敛所需的代数。
4。阅读 AlphaEvolve 的评估器设计笔记。为你关心的某个领域设计一个评估器（SQL 查询优化、测试套件最小化、部署 YAML）。
5。组合使用：用 HTN 把一个复合任务分解为子任务，再对每个子任务的原子操作符使用进化式搜索。它在哪里大放异彩，又在哪里过度工程化？

## 关键术语

| 术语 | 人们怎么说 | 它实际的含义 |
|------|----------------|------------------------|
| HTN | "层次规划器" | 带操作符、前置条件、效果的任务分解 |
| Method | "分解规则" | 将复合任务拆解为子任务的方式 |
| Operator | "原子动作" | 带前置条件和效果的具体步骤 |
| ChatHTN | "LLM + HTN" | 符号化规划器在没有方法匹配时询问 LLM |
| AlphaEvolve | "进化式代码搜索" | LLM 集成对代码进行突变；确定性评估器进行筛选 |
| Fitness function | "评估器" | 对输出进行确定性、机器可校验打分 |
| Online method learning | "缓存的 LLM 分解" | 存储并泛化 LLM 计划以削减查询成本 |
