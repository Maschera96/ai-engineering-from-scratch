给定一个仓库路径以及将在其中运行的智能体产品，审计七个 workbench 界面并生成一份就绪报告。

七个界面：

1. Instructions（指令）：智能体最先读取的根文件（例如 `AGENTS.md`），简短，并路由到更深层的规则。
2. State（状态）：一个持久、机器可读的文件，记录任务、改动过的文件、阻塞项、下一步动作。
3. Scope（范围）：每个任务的契约，列出允许的文件、禁止的文件、验收标准、回滚计划。
4. Feedback（反馈）：一个运行器，捕获命令、stdout、stderr、退出码，并把结果反馈回循环中。
5. Verification（验证）：一道关卡，运行测试、lint、类型检查、冒烟运行，并确认验收标准。
6. Review（评审）：由不同角色进行的第二轮检查，构建者不能给自己的工作打勾。
7. Handoff（交接）：一个工件，总结改了什么、为什么改、还剩什么、以及下一步最佳动作。

产出：

- 每个界面的评分：0 缺失，1 部分具备，2 健康。把每个评分绑定到你观察到的某个文件或流程。
- 按杠杆效应排序的三个优先级：哪个缺失的界面如果最先补上，能消除最多的失败模式。
- 一份机器可读的 `workbench_audit.json` 报告，外加一份人类可读的 `workbench_audit.md` 摘要。
- 针对最薄弱界面的起步补丁：把评分从 0 推进到 1 的最小文件改动。

硬性拒绝：

- 没有文件路径或流程引用的“健康”评分。没有证据的审计会腐烂。
- 单一合并的“agent config”界面。把多个界面合并在一起会掩盖任务出问题时究竟是哪个失败了。
- 因为测试慢而跳过验证。如果验证不在 workbench 上，构建者就会给自己批改作业。

拒绝规则：

- 如果仓库完全没有测试命令，拒绝给出验证评分，并将其作为阻塞性发现暴露出来。
- 如果仓库没有版本控制历史，拒绝给出交接评分，并将其作为阻塞性发现暴露出来。
- 如果智能体产品以 root 运行或拥有不受限的文件访问权限，在定义出沙箱或写入列表之前，拒绝给出范围评分。

输出结构：

```
workbench-audit/
├── workbench_audit.json
├── workbench_audit.md
├── patches/
│   └── <weakest-surface>.patch
└── README.md
```
