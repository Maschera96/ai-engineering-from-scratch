给定一个任务类别（重复出现的智能体运行类型——例如"重构一个函数""关闭一张支持工单"），维护一个反思的情景记忆缓冲区。每条反思用自然语言记录一种失败模式以及对应的纠正性洞见。该缓冲区会被前置到同一任务类别的下一次试验中。

产出：

1. 反思捕获。当一次试验以低于阈值的评估器分数结束后，输出一行形如"我未能做到 X，因为 Y；下次，Z"的反思。丢弃针对外部失败（网络、上游 500 错误）的反思，除非它们是可复现的。
2. TTL 与去重。反思默认在 N 次试验后过期（建议为 10）。完全相同的重复项会被合并。近似重复项（在小型嵌入模型上余弦相似度 >0.9，或共享子串 >= 80%）只保留最近的一条。
3. 作用域策略。三种作用域：task-class（按任务名称）、user（同一用户的跨任务）、agent（所有用户范围）。默认是 task-class。仅当反思涉及用户特定的偏好时才升级到 user 作用域；绝不自动升级到 agent 作用域。
4. 压实（Compaction）。当缓冲区超出预算时，运行 sleep-time 压实：聚类近似重复项、做总结、合并。压实在热路径之外运行——不要延迟主智能体的响应。
5. 提示集成。输出一个标题为"我从先前试验中学到的"的单一块，内含一个项目符号列表。在提示中上限为 6 项；超出部分归入一个单独的总结项（"……以及另外 4 条关于超时的较旧反思"）。

硬性拒绝：

- 把反思写成"下次更小心些"。那不可执行。用一个强制给出具体"下次"指令的提示重新运行反思器。
- 基于挂钟时间而非试验次数来让反思过期。为了离线可重放的运行，TTL 应以试验为作用域，而非以时间为作用域。
- 存储引用了密钥（API key、token、PII）的反思。在提交到缓冲区之前，以特定的"contains secret"类错误拒绝。

拒绝规则：

- 如果没有附加评估器，拒绝并推荐第 05 课（Self-Refine/CRITIC）——反思需要一个信号，而不是凭感觉。
- 如果任务类别是一次性的（永不重复），拒绝；情景记忆对一个永不重复的任务毫无用处。

输出：一个结构化的缓冲区文件（包含反思对象的 JSON：trial id、task class、scope、text、created_at、ttl_remaining）、一个用于下次试验的提示块，以及一份"过期反思"报告，列出即将过期的条目。

最后附上一条"接下来读什么"的提示：如果缓冲区持续触及上限，指向第 06 课（上下文压缩）；如果要把压实移出热路径，则指向第 08 课（Letta sleep-time compute）。
