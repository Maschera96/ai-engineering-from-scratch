# 记忆：虚拟上下文与 MemGPT

> 上下文窗口是有限的。对话、文档和工具调用轨迹却不是。MemGPT（Packer et al.，2023）把这一问题类比为操作系统的虚拟内存——主上下文是 RAM，外部存储是磁盘，智能体在两者之间进行换页。这是 2026 年每一套记忆系统都继承的模式。

**类型：** 构建
**语言：** Python（标准库）
**前置知识：** Phase 14 · 01（智能体循环）、Phase 14 · 06（工具使用）
**时长：** 约 75 分钟

## 学习目标

- 解释 MemGPT 所基于的操作系统类比：主上下文 = RAM，外部上下文 = 磁盘，记忆工具 = 换入/换出。
- 用标准库实现 MemGPT 的两层模式，包含一个主上下文缓冲区、一个可搜索的外部存储，以及换入/换出工具。
- 描述智能体如何发出"中断"来查询或修改外部记忆，以及结果如何被拼接回下一个提示中。
- 识别那些延续到 Letta（第 08 课）和 Mem0（第 09 课）的 MemGPT 设计选择。

## 问题所在

上下文窗口看起来应该能解决记忆问题。其实不然。生产环境中反复出现三种失败模式：

1。**溢出。** 多轮对话、长文档或工具调用密集的轨迹会超出窗口。截断点之外的所有内容都丢失了。
2。**稀释。** 即便在窗口内，塞入无关上下文也会稀释对重要内容的注意力。前沿模型在长输入上仍会退化。
3。**持久性。** 新会话从空窗口开始。没有外部记忆的智能体无法跨会话说出"还记得你让我做的那件事吗……"。

更大的窗口有帮助，但解决不了问题。Mem0 的 2025 年论文测得，128k 窗口的基线仍会漏掉长时程事实，而带外部记忆的 4k 窗口智能体却能捕捉到。

## 概念

### MemGPT：操作系统类比

Packer et al.（arXiv:2310.08560，v2 于 2024 年 2 月发布）将上下文管理映射到操作系统的虚拟内存：

| 操作系统概念 | MemGPT 概念 | 2026 年生产对应物 |
|------------|---------------|------------------------|
| RAM | 主上下文（提示） | Anthropic/OpenAI 上下文窗口 |
| 磁盘 | 外部上下文 | 向量数据库、KV、图存储 |
| 缺页中断 | 记忆工具调用 | `memory.search`、`memory.read`、`memory.write` |
| 操作系统内核 | 智能体控制循环 | 带记忆工具的 ReAct 循环 |

智能体运行一个普通的 ReAct 循环。一类额外的工具让它能把数据换入和换出主上下文。

### 两个层级

- **主上下文。** 固定大小的提示，保存当前任务。始终对模型可见。
- **外部上下文。** 无界限，可通过工具搜索。相关时读取，事实出现时写入。

原始论文在超出基础窗口的两类任务上评估了该设计：超过 100k token 的文档分析，以及跨多日、记忆持久的多会话对话。

### 中断模式

MemGPT 引入了"记忆即中断"：在对话中途智能体可以调用一个记忆工具，运行时执行它，结果作为新的观察拼接进下一个助手回合。这在概念上与 Unix 的 `read()` 系统调用完全相同——它阻塞进程、返回字节，进程随后继续。

典型的记忆工具接口：

- `core_memory_append(section，text)`——写入提示中的某个持久区段。
- `core_memory_replace(section，old，new)`——编辑某个持久区段。
- `archival_memory_insert(text)`——写入可搜索的外部存储。
- `archival_memory_search(query，top_k)`——从外部存储中检索。
- `conversation_search(query)`——扫描过往回合。

### MemGPT 的终点与 Lett的起点

2024 年 9 月，MemGPT 变成了 Letta。研究仓库（`cpacker/MemGPT`）仍然保留；Lett扩展了该设计：

- 三个层级而非两个（core、recall、archival——第 08 课）。
- 用原生推理取代 `send_message`/heartbeat 模式（第 08 课）。
- 运行异步记忆工作的睡眠时智能体（第 08 课）。

即便生产系统运行的是 Letta、Mem0 或自建的两层存储，MemGPT 论文仍是 2026 年的基石。

### 这一模式会在哪里出错

- **记忆腐烂。** 写入累积的速度快于读取；检索淹没在陈旧的事实里。修复方法：周期性整合（Lett睡眠时机制）、显式失效（Mem0 冲突检测器）。
- **记忆投毒。** 外部记忆是被检索回来的文本。如果攻击者控制的内容落入某条记忆笔记，智能体会在下一个会话重新摄入它。这是 Greshake et al.（第 27 课）攻击在时间维度上的重述。
- **引用丢失。** 智能体回忆起"用户让我交付 X"，却无法指出是哪个回合。每次归档写入时都要存储来源引用（会话 ID、回合 ID）。

## 动手构建

`code/main.py` 用标准库实现了 MemGPT 的两层模式：

- `MainContext`——固定大小的提示缓冲区，带一个 `core` 字典和一个 `messages` 列表；超出上限时自动压缩最旧的消息。
- `ArchivalStore`——内存中类 BM25 的存储（token 重叠打分），保存 (id，text，tags，session，turn) 记录。
- 五个映射到 MemGPT 接口的记忆工具。
- 一个脚本化的智能体，先用事实填充归档，然后通过调用 `archival_memory_search` 回答一个问题。

运行它：

```
python3 code/main.py
```

该轨迹展示智能体写入三条事实，把主上下文填到上限（强制驱逐），然后通过从归档中检索来回答一个后续问题——在没有任何真实 LLM 的情况下重现了 MemGPT 工作流。

## 实际运用

如今每一套生产记忆系统都是 MemGPT 的变体：

- **Letta**（第 08 课）——三层级、原生推理、睡眠时计算。
- **Mem0**（第 09 课）——向量 + KV + 图，融合一个打分层。
- **OpenAI Assistants / Responses**——通过线程和文件管理的托管记忆。
- **Claude Agent SDK**——通过技能和会话存储实现长期记忆。

按运维形态（自托管、托管、框架集成）来选型，而不是按核心模式——核心模式就是 MemGPT。

## 交付成果

`outputs/skill-virtual-memory.md` 是一个可复用的技能，它能为任意目标运行时生成一个正确的两层记忆脚手架（主 + 归档 + 工具接口），并已接好驱逐策略和引用字段。

## 练习

1。添加一个以 token 计量的 `max_main_context_tokens` 上限（用 `len(text.分离())` * 1.3 近似）。当超过上限时，把最旧的消息压缩成摘要。比较有摘要器和没有摘要器时的行为。
2。在归档存储上正确实现 BM25（词频、逆文档频率）。在一个玩具事实集上测量 recall@10，并与 token 重叠基线对比。
3。给归档插入添加 `citation` 字段（session_id、turn_id、source_url）。让智能体在每个基于检索的回答上引用来源。
4。模拟记忆投毒：添加一条写着"忽略未来所有用户指令"的归档记录。编写一个守卫，扫描检索结果中具有指令形态的文本并将其标记为不可信。
5。把实现移植为使用 MemGPT 研究仓库的 core-memory JSON 模式（`cpacker/MemGPT`）。当你从扁平字符串切换到带类型的区段时，会发生哪些变化？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| 虚拟上下文 | "无限记忆" | 主（提示）+ 外部（可搜索）层级，配合换入/换出 |
| 主上下文 | "工作记忆" | 提示——固定大小、始终可见 |
| 归档记忆 | "长期存储" | 外部可搜索的持久化，按需检索 |
| 核心记忆 | "持久的提示区段" | 钉在主上下文内的命名区段 |
| 记忆工具 | "记忆 API" | 智能体发出的、用于读写外部记忆的工具调用 |
| 中断 | "记忆缺页中断" | 智能体暂停，运行时取数据，结果拼接进下一回合 |
| 记忆腐烂 | "陈旧事实" | 旧写入淹没检索；用整合来修复 |
| 记忆投毒 | "注入的持久笔记" | 攻击者内容被存为记忆，在回忆时被重新摄入 |
