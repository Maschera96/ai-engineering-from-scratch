# 失败模式：智能体为何会崩溃

> MASFT（Berkeley，2025）将 14 种多智能体失败模式归入 3 个类别。Microsoft 的 Taxonomy 记录了现有 AI 失败如何在智能体场景中被放大。行业实地数据汇聚到五种反复出现的模式：幻觉动作、范围蔓延、级联错误、上下文丢失、工具误用。

**类型：** 学习 + 构建
**语言：** Python（stdlib）
**前置知识：** Phase 14 · 05（Self-Refine 与 CRITIC）、Phase 14 · 24（可观测性）
**时长：** 约 60 分钟

## 学习目标

- 说出 MASFT 的三个失败类别，以及每个类别中至少四种具体模式。
- 解释为什么智能体失败会放大现有的 AI 失败模式（偏见、幻觉）。
- 描述五种行业反复出现的模式及其缓解方法。
- 实现一个 stdlib 检测器，为智能体轨迹打上失败模式标签。

## 问题

团队交付的智能体在 90% 的轨迹上能正常工作。那 10% 的失败并不是随机噪声——它们落入少数几个反复出现的类别。一旦你能为它们命名，就能对其进行监控并修复。

## 概念

### MASFT（Berkeley，arXiv:2503.13657）

Multi-Agent System Failure Taxonomy（多智能体系统失败分类）。14 种失败模式聚为 3 个类别。标注者间一致性 Cohen's Kappa 为 0.88——这些类别可以可靠地区分开。

核心论点：失败是多智能体系统中根本性的设计缺陷，而不是需要用更好的基座模型来修复的 LLM 局限。

### Microsoft Taxonomy of Failure Mode in Agentic AI Systems

- 现有 AI 失败（偏见、幻觉、数据泄露）在智能体场景中被放大。
- 自主性带来了新的失败：大规模的非预期动作、工具误用、任务漂移。
- 这份白皮书就是智能体产品的风险登记册。

### Characterizing Faults in Agentic AI（arXiv:2603.06847）

- 失败源于编排、内部状态演化以及环境交互。
- 不只是“糟糕的代码”或“糟糕的模型输出”。

### LLM Agent Hallucinations Survey（arXiv:2509.18970）

两种主要表现：

1. **指令遵循偏离（Instruction-following Deviation）**——智能体不遵循系统提示。
2. **长程上下文误用（Long-range Contextual Misuse）**——智能体遗忘或错误应用更早轮次的上下文。

子意图错误：遗漏（Omission，漏掉一步）、冗余（Redundancy，重复一步）、错序（Disorder，步骤顺序错乱）。

### 五种行业反复出现的模式

Arize、Galileo、NimbleBrain 在 2024–2026 年的实地分析汇聚到：

1. **幻觉动作。** 智能体调用了一个不存在的工具，或者编造了参数。
2. **范围蔓延。** 智能体把任务扩展到用户请求之外（创建额外的 PR、发送额外的邮件）。
3. **级联错误。** 一次错误调用触发了下游连锁反应。一个幻觉出来的虚假 SKU 触发了四次 API 调用——演变成多系统事故。
4. **上下文丢失。** 长周期任务遗忘了早期轮次的约束。
5. **工具误用。** 用错误的参数调用了正确的工具，或者干脆调用了错误的工具。

级联是致命的。智能体无法区分“我失败了”和“这个任务不可能完成”，并且经常在收到 400 错误时幻觉出一条成功消息来闭合循环。

### 缓解：每一步都设关卡

在推理链的每一步设置自动化验证关卡，对照环境状态检查事实依据。具体而言：

- 每步安全分类器（第 21 课）。
- 工具调用参数校验（第 06 课）。
- 将检索到的内容与已知事实交叉核对（第 05 课，CRITIC）。
- 通过重新探查状态来检测成功幻觉（文件真的被创建了吗？）。

### 失败监控容易出错的地方

- **只标记崩溃。** 大多数智能体失败会产生看似有效的输出。需要内容层面的检查。
- **没有基线。** 漂移检测需要一个“最近已知良好”状态；没有它，你无法判断“这正在变糟”。
- **过度告警。** 每次失败都触发一次寻呼。要做聚类和限流。

## 动手构建

`code/main.py` 实现了一个 stdlib 失败模式标注器：

- 一个覆盖五种模式的合成轨迹数据集。
- 每种模式对应的检测器函数（针对工具调用、输出、重复动作的特征模式）。
- 一个标注器，为每条轨迹打标签并报告模式分布。

运行它：

```
python3 code/main.py
```

输出：每条轨迹的标签 + 聚合分布，是对 Phoenix 轨迹聚类所揭示内容的一个廉价复现。

## 使用它

- **Phoenix** 用于生产环境的漂移聚类（第 24 课）。
- **Langfuse** 用于会话回放 + 标注。
- **自建** 用于你的可观测性平台无法检测的领域特定特征。

## 交付它

`outputs/skill-failure-detector.md` 生成针对你所在领域定制的失败模式检测器，并接入轨迹存储。

## 练习

1. 添加一个针对“成功幻觉”的检测器：智能体返回成功，但目标状态并未改变。
2. 为你构建过的某个产品标注 100 条真实轨迹。哪种模式占主导？修复它的成本是多少？
3. 实现一个“级联半径”指标：给定第 N 步的某次失败，它影响了多少个下游步骤？
4. 阅读 MASFT 的 14 种失败模式。挑出三种适用于你产品的模式。编写检测器。
5. 将一个检测器接入 CI 作业：如果 >=5% 的轨迹被标记为某种模式，则让构建失败。

## 关键术语

| 术语 | 人们怎么说 | 它实际指什么 |
|------|----------------|------------------------|
| MASFT | “多智能体失败分类” | Berkeley 的 14 模式归类 |
| 级联错误 | “涟漪失败” | 一个早期错误穿过 N 个步骤传播 |
| 上下文丢失 | “忘了约束” | 长周期轮次丢弃了早期轮次的事实 |
| 工具误用 | “错误的工具 / 错误的参数” | 调用有效，但调用方式错误 |
| 成功幻觉 | “伪造完成” | 智能体在收到 400 时声称成功；状态未改变 |
| 范围蔓延 | “越界” | 智能体做了超出要求的事 |
| 指令遵循偏离 | “违抗” | 忽略系统提示或用户约束 |
| 子意图错误 | “计划缺陷” | 计划执行中的遗漏、冗余、错序 |
