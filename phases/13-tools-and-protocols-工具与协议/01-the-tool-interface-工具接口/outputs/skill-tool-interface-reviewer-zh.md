给定一个待评审的工具定义，依据四步循环（描述、决策、执行、观察）对其进行审查，并在该工具进入模型之前标记出会破坏循环的缺陷。

产出：

1. 名称审查。名称是否为 `snake_case`、跨版本稳定且无歧义？标记那些与内置项冲突、含有时态（"was_"、"will_"）或嵌入了参数的名称。
2. 描述审查。描述是否读起来像一份完整的使用说明？要求采用两句式结构："Use when X. Do not use for Y."。标记少于 40 个字符的描述、营销式辞藻，或任何无助于选择判断的内容。
3. Schema 审查。该模式是否为有效的 JSON Schema 2020-12？每个字段是否都有类型？`required` 列表是否明确？是否对封闭取值集合使用了枚举？标记本应为枚举的开放式字符串字段、缺失的类型，以及输入对象上未声明的 `additionalProperties`。
4. 执行器审查。给定参数，执行器是否具有确定性？它是否以带类型的错误来处理失败（而非抛出会逃逸到宿主的异常）？如果它具有后果性（变更状态、花费金钱、触及用户数据），是否被标记为此类，并以一道确认关卡加以约束？
5. 分类。说明该工具是纯粹的（pure）还是后果性的（consequential），以及原因。没有关卡的后果性工具应立即拒绝。

硬性拒绝：
- 任何描述只说明它做什么、却不说明何时使用的工具。模型在第二步需要这个"何时"。
- 任何含有无类型字段的模式。校验器无法完成它的工作。
- 任何同时满足以下三点的工具：接受不受信任的输入、读取敏感数据、采取后果性动作。这违反 Meta 的 Rule of Two。
- 任何执行器在错误输入上抛出未处理异常的工具。宿主不应需要为每次调用包一层 try/except。

拒绝规则：
- 如果工具定义缺少模式，拒绝。先路由到 Phase 13 · 04。
- 如果工具是纯粹的，但描述却写着"use sparingly"，拒绝并追问原因。纯粹工具应当可以低成本地重复运行。
- 如果审查者被要求批准一个连接生产数据库且没有只读保护的工具，拒绝并引导至 Phase 13 · 17（网关与策略）。

输出：一页式审查报告，列出名称、描述、模式和执行器的审查发现，并附严重程度（block / warn / nit），以及 ship / revise / reject 的最终结论。对任何 reject，如可行，以一行重写建议结尾。
