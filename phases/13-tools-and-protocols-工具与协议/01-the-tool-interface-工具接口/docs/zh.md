# 工具接口 —— 为什么智能体需要结构化的 I/O

> 语言模型产出 token，程序执行动作。这两者之间的鸿沟就是工具接口：一份让模型可以请求动作、由宿主执行动作的契约。2026 年的每一套技术栈 —— OpenAI、Anthropic 和 Gemini 上的函数调用；MCP 的 `tools/call`；A2A 的任务分片 —— 都是同一个四步循环的不同编码方式。本课为这个循环命名，并展示运行它所需的最小机制。

**类型：** 学习
**语言：** Python（标准库，无 LLM）
**前置：** 第 11 阶段（LLM 补全 API）
**时长：** 约 45 分钟

## 学习目标

- 解释为什么一个只能生成文本的 LLM 无法独自对现实世界采取动作。
- 画出四步工具调用循环（描述 → 决定 → 执行 → 观察），并指出每一步由谁负责。
- 将工具描述写成三个部分：名称、JSON Schema 输入，以及一个确定性的执行器函数。
- 区分纯工具和有副作用的工具，并说明这种划分为什么对安全很重要。

## 问题所在

LLM 输出的是下一个 token 的概率分布。这就是它全部的输出表面。如果你问聊天模型「班加罗尔现在的天气怎么样」，它能写出一个看似合理的句子，但它没法接入天气 API。这个句子可能碰巧正确，也可能是三天前的过时信息。

弥合这一鸿沟正是工具接口的目的。宿主程序 —— 你的智能体运行时、Claude Desktop、ChatGPT、Cursor 或一个自定义脚本 —— 向模型公布一份可调用的工具清单。模型在判断需要采取动作时，会发出一份结构化载荷，指明某个工具及其参数。宿主解析这份载荷，真正运行该工具，并把结果回喂给模型。循环持续进行，直到模型判断不再需要调用为止。

这份契约的第一个版本于 2023 年 6 月作为 OpenAI 的 "functions" 参数发布。Anthropic 随后在 Claude 2.1 中推出了 `tool_use` 块。Gemini 几个月后加入了 `functionDeclarations`。如今每家供应商都暴露相同的形态：输入一份用 JSON Schema 标注类型的工具清单，输出一个 JSON 载荷工具调用。Model Context Protocol（2024 年 11 月）将这份契约通用化，使得一个工具注册表可以服务于每一个模型。A2A（2026 年 4 月，v1.0）在同一个原语之上叠加了智能体之间的委派。

四步循环是所有这些方案底层的不变量。第 13 阶段的其余一切都是对它的展开。

## 概念

### 第一步：描述

宿主用三个字段声明每个工具。

- **名称（Name）。** 一个稳定、机器可读的标识符。是 `get_weather`，而不是「天气那东西」。
- **描述（Description）。** 一段自然语言简述。「当用户询问某个特定城市的当前状况时使用。不要用于历史数据。」
- **输入模式（Input schema）。** 一个描述工具参数的 JSON Schema 对象（draft 2020-12）。

模型接收这份清单。现代供应商会用各自特定的模板把这些声明序列化进系统提示词，所以作为调用方的你只需处理结构化的形式。

### 第二步：决定

给定用户的消息和可用工具，模型会选择以下三种行为之一。

1. **直接用文本回答。** 不调用工具。
2. **调用一个或多个工具。** 发出结构化的调用对象。在 `parallel_tool_calls: true` 下（OpenAI 和 Gemini 默认开启，Anthropic 需主动启用），模型可以在一个回合内发出多个调用。
3. **拒绝。** 严格模式的结构化输出可以产出一个有类型的 `refusal` 块，而不是一次调用。

一份工具调用载荷有三个稳定字段：一个调用 `id`、一个工具 `name`，以及一个 JSON `arguments` 对象。id 的存在使宿主能够将稍后的结果与具体的调用关联起来,这在并行调用乱序返回时尤为重要。

### 第三步：执行

宿主接收调用，根据声明的模式校验参数，然后运行执行器。无效参数意味着模型臆造了某个字段或用错了类型 —— 这在弱模型上是非常常见的失败模式。生产级宿主在遇到无效参数时会做三件事之一：快速失败并把错误暴露给模型，用受约束的解析器修复 JSON,或者把校验错误一并放进提示词重试模型。

执行器本身就是普通代码。Python、TypeScript、一条 shell 命令、一次数据库查询。它产出一个结果,通常是字符串,但也可以是任意 JSON 值或结构化内容块（在 MCP 中是文本、图片或资源引用）。结果必须是可序列化的。

### 第四步：观察

宿主把工具结果追加到对话中（作为带有匹配 `id` 的 `tool` 角色消息），并重新调用模型。模型现在上下文中有了工具输出,可以产出最终答案或请求更多调用。这会持续进行,直到模型停止发出调用,或宿主触及迭代次数的安全上限。

### 信任划分

工具有两种对安全很重要的类型。

- **纯工具（Pure）。** 只读、确定性、无副作用。`get_weather`、`search_docs`、`get_current_time`。可以放心地推测性调用。
- **有后果的工具（Consequential）。** 会改变状态、花钱、触及用户数据。`send_email`、`delete_file`、`execute_trade`。必须加门控。

Meta 在 2026 年提出的智能体安全「二选规则（Rule of Two）」指出,单个回合最多只能组合以下三者中的两者：不可信输入、敏感数据、有后果的动作。工具接口正是你执行这条规则的地方 —— 通过拒绝调用、要求用户确认或提升权限范围。完整的安全章节见第 13 阶段 · 15,智能体层级的权限策略见第 14 阶段 · 09。

### 循环存在于何处

| 场景 | 谁负责描述 | 谁负责决定 | 谁负责执行 |
|---------|---------------|-------------|--------------|
| 单回合函数调用（OpenAI/Anthropic/Gemini） | 应用开发者 | LLM | 应用开发者 |
| MCP | MCP 服务器 | 经由 MCP 客户端的 LLM | MCP 服务器 |
| A2A | Agent Card 发布者 | 调用方智能体 | 被调用方智能体 |
| Web 浏览器（函数调用智能体） | 浏览器扩展 / WebMCP | LLM | 浏览器运行时 |

处处都是同样的四步。列名在变,结构不变。

### 为什么不直接提示模型输出 JSON？

「让模型用 JSON 回复」是函数调用出现之前的模式。它在前沿模型上有约 5% 到 15% 的失败率,在较小的模型上失败率高得多。失败模式包括缺失花括号、尾随逗号、臆造字段和错误类型。然后你还需要一道 JSON 修复流程、一次重试,或一个受约束的解码器。

原生函数调用更好,原因有三。第一,供应商在精确的调用形态上端到端训练模型,所以在严格模式下有效 JSON 的比率攀升到 98% 到 99%。第二,调用载荷位于它自己的协议槽位中,而不在自由文本内部 —— 因此工具调用绝不会泄漏进用户可见的回复里。第三,供应商用受约束的解码来强制模式合规（OpenAI 的严格模式、Anthropic 的 `tool_use`、Gemini 的 `responseSchema`）。输出保证能通过校验。

### 断路器

当模型停止发出调用,或宿主触及最大回合数时,循环终止。生产级宿主会把这个值设在 5 到 20 个回合之间。超过这个范围,你几乎肯定陷入了一个模型无法退出的循环。Claude Code 默认为 20;OpenAI Assistants 为 10;Cursor 的智能体模式为 25。

另一种做法 —— 无界循环 —— 每隔六个月就会以「智能体一夜之间花掉了 400 美元 API 调用费」的事后复盘形式出现。没有上限就不要上线。

## 上手用

`code/main.py` 在没有 LLM 的情况下运行这个四步循环。一个伪造的「决策者」函数通过对用户消息做模式匹配来模拟模型;而执行器、模式校验器和观察步骤的脚手架都是真实的。运行它,你能看到完整的请求/响应编排,带有可打印的中间状态,然后在后面的课程里把伪造的决策者换成任意真实供应商。

需要关注的地方：

- 工具注册表为每个工具保存了三个字段：名称、描述、模式,以及一个执行器引用。
- 校验器是一个最小化的 JSON Schema 子集（类型、required、enum、min/max）,仅用标准库编写。第 13 阶段 · 04 会交付一个更完整的版本。
- 循环把迭代次数限制在五次。生产级智能体需要的正是这种断路器。

## 交付物

本课产出 `outputs/skill-tool-interface-reviewer.md`。给定一份工具定义草稿（名称 + 描述 + 模式 + 执行器大纲）,该技能会审计它是否适配循环：名称是否机器稳定,描述是否是一份完整的用法简述,模式是否正确使用了 JSON Schema 2020-12,以及纯工具与有后果工具的分类是否明确。

## 练习

1. 在 `code/main.py` 中新增第四个工具,名为 `get_stock_price(ticker)`。把它的描述写成「当用户按股票代码询问当前股价时使用。不要用于历史价格或市场摘要。」运行脚手架,确认伪造的决策者会把提及股票代码的查询路由到这个新工具。

2. 弄坏模式校验器。传入一个 `arguments` 对象缺失某个必填字段的调用,确认宿主在执行前就拒绝它。然后传入一个带有多余未知字段的调用。做出决定：宿主应该拒绝还是忽略?用一条安全论据为你的选择辩护。

3. 把脚手架中的每个工具分类为纯工具或有后果的工具。给需要的注册表条目加上 `consequential: true` 标志,并修改循环,使得每当选中一个有后果的工具时,打印一行「将与用户确认」。这就是每个生产级宿主都需要的确认门控的形态。

4. 在纸上画出四步循环,并为你最喜欢的客户端（Claude Desktop、Cursor、ChatGPT 或一个自定义技术栈）填好上面的供应商列表。与第 13 阶段 · 06 中 MCP 专属的变体做交叉对照。

5. 从头到尾读一遍 OpenAI 的函数调用指南。找出那个位于请求中、但不在本课所呈现的四步循环里的字段。解释它增加了什么,以及它为何是便利的而非必需的。

## 关键术语

| 术语 | 人们怎么说 | 它实际指什么 |
|------|----------------|------------------------|
| 工具（Tool） | 「模型可以调用的东西」 | 名称 + JSON Schema 类型化输入 + 执行器函数 的三元组 |
| 函数调用（Function calling） | 「原生工具使用」 | 供应商层级的 API 支持,用于发出结构化工具调用而非散文 |
| 工具调用（Tool call） | 「模型发出的动作请求」 | 由模型发出的、带有 `id`、`name`、`arguments` 的 JSON 载荷 |
| 工具结果（Tool result） | 「工具返回的东西」 | 执行器的输出,包装在带有匹配 id 的 `tool` 角色消息中 |
| 并行工具调用（Parallel tool calls） | 「一次多个调用」 | 一个模型回合内的多个调用对象,相互独立且可按 id 排序 |
| 严格模式（Strict mode） | 「保证的 JSON」 | 受约束的解码,强制模型的输出通过声明模式的校验 |
| 纯工具（Pure tool） | 「只读工具」 | 无副作用;可安全重新运行 |
| 有后果的工具（Consequential tool） | 「动作工具」 | 改变外部状态;需要门控、审计或用户确认 |
| 四步循环（Four-step loop） | 「工具调用周期」 | 描述 → 决定 → 执行 → 观察 |
| 宿主（Host） | 「智能体运行时」 | 持有工具注册表、调用模型并运行执行器的程序 |
