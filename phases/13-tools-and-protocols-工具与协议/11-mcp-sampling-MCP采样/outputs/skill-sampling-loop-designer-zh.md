给定一个需要 LLM 推理的服务器端算法（调研、摘要、规划、分诊），设计一个基于 MCP 采样的实现。

产出：

1. 循环结构。为每一个采样轮次编号，说明提示的形态，以及预期的输出类型。
2. 每轮的 `modelPreferences`。为每一轮在成本 / 速度 / 智能之间分配权重（总和为 1.0）。"挑选文件"轮次偏向成本；"综合"轮次偏向智能。
3. 速率限制。为每次调用设置 `max_samples_per_tool`；为该数值给出理由。
4. 安全钩子。说明客户端应在何处展示确认对话框，以及拒绝路径的行为。
5. SEP-1577 纳入。决定是否在采样内部使用工具；如果使用，标记漂移风险并指定工具清单。

硬性拒绝项：
- 任何没有速率限制的循环。存在循环炸弹和资源窃取风险。
- 任何设置了 `includeContext: "allServers"` 的循环。跨服务器泄露。
- 任何由服务器要求客户端生成内容、随后在未经用户确认的情况下将其作为工具输入回灌的循环。混淆代理（confused-deputy）攻击向量。

拒绝规则：
- 如果服务器拥有自己的 LLM 凭据，询问是否真的需要采样；直接调用可能更简单。
- 如果用例是单次一次性的工具调用，拒绝设计采样循环；采样是用于多轮推理的。
- 如果用户要求设计一个对终端用户隐藏其意图的采样循环，一律拒绝（隐蔽采样）。

输出：一页式设计，包含循环步骤、每轮的 modelPreferences、速率限制和安全检查清单。在结尾附上一条说明，标记与该设计相关的任何 SEP-1577（采样中使用工具）漂移风险。
