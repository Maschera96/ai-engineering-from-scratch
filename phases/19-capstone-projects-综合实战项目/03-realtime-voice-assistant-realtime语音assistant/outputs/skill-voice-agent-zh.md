给定一个域（客户支持、调度、零售助理），部署一个 WebRTC 语音代理，将端到端首次音频输出保持在 800 毫秒以下，同时处理插入、工具调用和数据包丢失。
建设计划：
1. 建立一个 LiveKit Agents 1.0 房间，其中包含可传输麦克风音频的 Web 客户端。添加 Twilio PSTN 网关以实现电话覆盖。
2. 运行流式 ASR（Deepgram Nova-3 托管或 g5.xlarge 上的 fast-whisper Whisper-v3-turbo）。订阅部分和最终成绩单。
3. 在 20 毫秒帧上运行 Silero VAD v5。在演讲结束时，使用 LiveKit 转弯检测器对最新的部分进行评分；仅当 VAD 静默 >= 500ms 且完成分数 >= 0.6 时才承诺完成回合。
4. 流式传输 LLM（GPT-4o-realtime、Gemini 2.5 Flash Live 或级联 Claude Haiku 4.5）。在 200 毫秒内将第一个令牌交给 TTS。
5. 流式传输 TTS（Cartesia Sonic-2 或 ElevenLabs Flash v3）。第一个音频块必须在第一个 LLM 令牌后 200 毫秒内离开服务器。
6. 打断：当 VAD 在 SPEAKING 或 THINKING 期间检测到新的用户语音时，取消 TTS，丢弃剩余的 LLM 输出，重新启用 ASR。发布 `tts_canceled` 跨度。
7.工具侧通道：并发运行函数调用；如果延迟 > 300 毫秒，则发出确认填充符，以便音频流永远不会停止。
8. 记录100个通话。根据保留的转录本、Hamming VAD 基准的错误截止率、首个音频输出 p50、NISQA MOS 以及 3% 丢包率下的行为来测量 WER。
9. 使用合成调用者对单个 g5.xlarge 上的 50 个并发调用进行负载测试；报告持续第一音频输出p95。
评估标准：
|重量 |标准|测量|
|:-:|---|---|
| 25 | 25端到端延迟| p50 在 100 个录音通话中首次音频输出时间低于 800 毫秒 |
| 20 |轮流品质 | Hamming VAD 基准的错误截断率低于 3% |
| 20 |工具使用正确性 |对话中工具调用返回正确的数据，而不会停止音频 |
| 20 |丢包情况下的可靠性 |注入 3% 丢包后的 WER 和轮流稳定性 |
| 15 | 15评估线束完整性|使用公共配置进行可重复测量 |
硬拒绝：
- 非流管道（批处理 ASR、批处理 TTS）无法达到延迟目标。
- 任何不立即取消 TTS 缓冲区的打断策略。延迟取消会产生最糟糕的用户体验回归。
- 同步阻止 LLM 流的工具调用。他们必须在侧通道上运行。
拒绝规则：
- 拒绝在没有 VAD 或转弯检测器的情况下部署。固定超时轮流产生不可接受的截止率。
- 拒绝报告 MOS，而不记录其是否经过人工评级或 NISQA 代理。
- 在没有至少 100 条通话记录并发布通话痕迹的情况下，拒绝报告“X 下的 p50 延迟”。
输出：包含 LiveKit 代理工作程序、PSTN 网关配置、100 次呼叫评估工具、公共 Langfuse 语音仪表板、与一个托管竞争对手（直接 Retell、Vapi 或 OpenAI Realtime API）的并排比较，以及关于您观察到的三个最大轮流故障以及修复每个故障的检测器调整的记录。
