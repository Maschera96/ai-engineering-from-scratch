# 任意分辨率视觉：Patch-n'-Pack 与 NaFlex

> 真实图像并不是 224x224 的正方形。一张收据是 9:16，一张图表是 16:9，一张医学扫描可能是 4096x4096，一张手机截图是 9:19.5。2024 年之前 VLM 的做法——把一切都缩放成固定正方形——丢弃了让 OCR、文档理解和高分辨率场景解析得以工作的信号。NaViT（Google，2023）展示了你可以用块对角掩码把可变分辨率的补丁打包进单个 transformer 批次。Qwen2-VL 的 M-RoPE（2024）彻底抛弃了绝对位置表。LLaVA-NeXT 的 AnyRes 把高分辨率图像切成基础图 + 子图。SigLIP 2 的 NaFlex 变体（2025）如今成了那些希望用单个检查点服务所有宽高比的开源 VLM 的默认编码器。本课从头到尾实现 patch-n'-pack。

**类型：** 构建
**语言：** Python（标准库，补丁打包器 + 块对角掩码）
**前置：** Phase 12 · 01（ViT 补丁）、Phase 12 · 05（LLaVA）
**时间：** 约 120 分钟

## 学习目标

- 把一批可变分辨率图像的补丁打包成一个序列，并构建块对角注意力掩码。
- 为给定任务在 AnyRes 切片（LLaVA-NeXT）、NaFlex（SigLIP 2）和 M-RoPE（Qwen2-VL）之间做选择。
- 在不缩放的情况下为 OCR、图表和摄影计算 token 预算。
- 说出正方形缩放的三种失败模式：被挤压的文本、被裁剪的内容、浪费在填充上的 token。

## 问题

Transformer 期待一个序列。一个批次是若干长度相同的序列堆叠。如果你的图像是 224x224，每次都得到 196 个补丁 token，无需填充，搞定。在 224 上训练，在 224 上推理，再也不用考虑分辨率。

但世界并不配合。文档是纵向的（8.5x11 英寸，大约 2:3）。图表截图是横向的（16:9）。收据又高又窄（1:3）。医学影像以 2048x2048 或更大的尺寸交付。手机设备截图是 1170x2532（0.46:1）。

2024 年之前的三个选项以及它们各自为何失败：

1. 缩放成固定正方形（224x224 或 336x336）。挤压会扭曲文本和人脸。降采样会破坏图表标签和 OCR 内容。在 LLaVA-1.5 之前这是标准做法。
2. 裁剪成固定宽高比。你丢弃了图像的大部分，而选择裁剪位置本身又是一个视觉问题。
3. 填充到最长边。修复了扭曲，但纵向图像有 50% 以上的 token 浪费在填充上。在所有这些填充 token 上还有二次方的注意力代价。

2024-2025 年的答案：让 transformer 以图像的原生分辨率吃下补丁，并想办法把异构批次打包进一个序列而不浪费算力。

## 概念

### NaViT 与 patch-n'-pack

NaViT（Dehghani 等人，2023）是展示这在大规模下可行的论文。这个思路是机械式的：

1. 对批次中的每张图像，以选定的补丁大小（比如 14）计算其原生补丁网格。
2. 把每张图像的补丁展平成它自己的可变长度序列。
3. 把所有图像的补丁拼接成该批次的一个长序列。
4. 构建块对角注意力掩码，使图像 A 的补丁只在图像 A 内部进行注意力计算。
5. 携带每个补丁的位置信息（2D RoPE 或分数位置嵌入）。

一批三张图像——336x336（576 token）、224x224（256 token）和 448x336（768 token）——变成一个 1600 token 的序列，配一个 1600x1600 的块对角掩码。没有填充。没有浪费的算力。Transformer 处理任意宽高比。

NaViT 还引入了训练时的分数补丁丢弃——在整个批次上随机丢弃 50% 的补丁——它既起到正则化作用又加速训练。SigLIP 2 继承了这一点。

### AnyRes（LLaVA-NeXT）

LLaVA-NeXT 的 AnyRes 是务实的替代方案。给定一张高分辨率图像和一个固定编码器（CLIP 或 SigLIP，336），对图像切片：

1. 从一组预定义布局——(1x1)、(1x2)、(2x1)、(1x3)、(3x1)、(2x2) 等——中挑选最适合该图像宽高比的网格布局。
2. 把整张图像切成网格；每个切片成为一张 336x336 的裁剪图。
3. 同时生成一张缩略图：整张图像缩放成 336x336，作为全局上下文 token。
4. 把每个切片送过冻结的 336 编码器。拼接切片 token + 缩略图 token。

对于一张 672x672 图像，采用 2x2 网格加缩略图：4 * 576 + 576 = 2880 个视觉 token。代价高但有效——LLM 同时看到局部细节和全局上下文。

当你的编码器被冻结且只支持单一分辨率时，AnyRes 是首选路线。它会让大图像的 token 数量爆炸（一张 1344x1344 图像在 4x4 网格下是 9216 + 576 ≈ 9800 token，足以填满大部分 8k LLM 的上下文）。

### M-RoPE（Qwen2-VL）

Qwen2-VL 引入了多模态旋转位置嵌入（Multimodal Rotary Position Embedding）。它不用 NaViT 的分数位置或 AnyRes 的切片加缩略图，而是让每个补丁携带一个 3D 位置（时间、高度、宽度）。查询/键的旋转处理任意的 H、W 和时间长度。

M-RoPE 原生支持动态分辨率，无需重新训练。推理时你输入任意 HxW 图像，补丁嵌入器产生 H/14 x W/14 个 token，每个 token 获得它的 (t=0, r=行, c=列) 位置，RoPE 用正确的频率旋转注意力，搞定。Qwen2.5-VL 和 Qwen3-VL 延续了这一点。InternVL3 的 V2PE 是同样的思路，但对每种模态采用可变编码。

与 AnyRes 不同，M-RoPE 在原生分辨率下是 O(H x W / P^2) 个 token——没有切片带来的乘性开销。与 NaViT 不同，它仍然期待每次前向传播只有单张图像。跨分辨率的批处理仍需要在其上叠加 patch-n'-pack。

### NaFlex（SigLIP 2）

NaFlex 是 SigLIP 2 检查点的原生柔性（native-flex）模式。单个模型在推理时服务多种序列长度（256、729、1024 token）。其内部在训练时使用 NaViT 风格的 patch-n'-pack，并对每个补丁采用绝对分数位置。卖点在于：一个检查点，在推理时根据任务挑选你的 token 预算。

对语义任务（分类、检索）用 256 token。对 OCR 或图表理解用 1024 token。无需重新训练。

### 打包掩码

块对角掩码是大多数实现栽跟头的地方。对于一个覆盖图像 `i=0..B-1`、长度为 `n_i`、总长度为 `N_total` 的打包序列，形状为 `(N_total, N_total)` 的掩码 `M`：当两个索引都落在同一张图像的块内时为 1，否则为 0。你可以从累积长度列表构建它：

```
offsets = [0, n_0, n_0+n_1, ..., N_total]
M[i, j] = 1 iff there exists b where offsets[b] <= i < offsets[b+1] and offsets[b] <= j < offsets[b+1]
```

在 PyTorch 中用 `torch.block_diag` 或显式 gather 即可一行搞定。FlashAttention 的可变长度路径（`cu_seqlens`）完全跳过掩码，直接用累积长度张量在各序列内部做注意力——对典型批次而言比稠密掩码快约 10 倍。

### Token 预算

按任务挑选你的策略：

- OCR / 文档：1024-4096 token。SigLIP 2 NaFlex 用 1024，或 AnyRes 3x3 + 缩略图。
- 图表与 UI：在 384-448 原生分辨率下用 729-1024 token。Qwen2.5-VL 动态分辨率配最大像素上限。
- 自然照片：256-576 token 即可。下游 LLM 看到的足够了。在内容密度高的地方才为 token 付费。
- 视频：空间池化后每帧 64-128 token，2-8 FPS。第 12.17 课讲这个。

2026 年的生产法则：为每个任务挑选一个最大像素上限，在该上限内以原生宽高比编码，打包批次，跳过填充。Qwen2.5-VL 暴露了 `min_pixels` 和 `max_pixels` 正是为这个旋钮。

## 动手用

`code/main.py` 为一批异构图像（带整数像素坐标）实现了 patch-n'-pack。它：

- 接收一个 (H, W) 图像尺寸列表。
- 在补丁大小 14 下计算每张图像的补丁序列长度。
- 把它们打包进一个总长度为 `sum(n_i)` 的序列。
- 构建块对角注意力掩码（稠密形式，便于理解）。
- 对比打包代价与正方形缩放、AnyRes 切片的代价。
- 为一个混合批次（收据、图表、截图、照片）打印 token 预算表。

运行它。最后跳出来的数字，正是 2026 年每个开源 VLM 都使用 patch-n'-pack 的原因。

## 交付

本课产出 `outputs/skill-resolution-budget-planner.md`。给定一个混合宽高比的工作负载（OCR、图表、照片、视频帧）和一个总 token 预算，它挑选正确的策略（NaFlex、AnyRes、M-RoPE 或固定正方形）并发出每请求的配置。当你在为产品规划 VLM 规模时使用这个技能——它能防止那种悄无声息地把延迟预算击垮的 10 倍 token 爆炸。

## 练习

1. 一张收据是 600x1500（1:2.5）。在补丁大小 14 下有多少原生分辨率 token？正方形缩放到 336 之后有多少？实践中哪个损失更多 OCR 准确率？

2. 为一批四张图像（长度分别为 256、576、729、1024）构建块对角掩码。验证注意力矩阵是 2585x2585，且恰好有 `256^2 + 576^2 + 729^2 + 1024^2` 个非零项。

3. 对一张 1792x896 图像、补丁 14，对比：(a) 正方形缩放到 336 后编码，(b) AnyRes 2x1 + 缩略图，(c) M-RoPE 原生分辨率。哪个用的 token 最少？哪个保留的细节最多？

4. 实现分数补丁丢弃：给定一个打包序列，均匀随机丢弃 50% 的 token，并相应更新块对角掩码。测量掩码的稀疏度变化。

5. 阅读 Qwen2-VL 论文（arXiv:2409.12191）的第 3.2 节。用两句话描述 `min_pixels` 和 `max_pixels` 控制什么，以及为何两个边界都重要。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------------|------------------------|
| Patch-n'-pack | "NaViT 风格的打包" | 把来自不同图像的可变长度补丁序列拼接进一个批次维度 |
| 块对角掩码 | "打包掩码" | 把每张图像的补丁限制为只对自身做注意力、不对打包中的邻居做注意力的注意力掩码 |
| AnyRes | "LLaVA-NeXT 切片" | 把高分辨率图像切成固定大小切片的网格加一张全局缩略图；用固定编码器编码每个切片 |
| NaFlex | "SigLIP 2 原生柔性" | 单个 SigLIP 2 检查点，在推理时服务 256/729/1024 token 的预算，无需重新训练 |
| M-RoPE | "多模态 RoPE" | 3D 旋转位置编码（时间、行、列），无需位置表即可处理任意 H、W、T |
| cu_seqlens | "FlashAttention 打包" | FlashAttention varlen 路径使用的累积长度张量，用来取代稠密块对角掩码 |
| 视觉 token 预算 | "每张图像多少 token" | 每张图像产出的补丁 token 的粗略计数；决定 LLM 的提示预算和注意力代价 |
| min_pixels / max_pixels | "分辨率边界" | Qwen2.5-VL 的每请求旋钮，对极小或极大输入的 token 数量设上限 |
