# 音频生成

> 音频是 16-48 kHz 的一维信号。 5 秒的剪辑包含 80-240k 个样本。没有变压器直接参与该序列。 2026 年每个生产音频模型的解决方案都是相同的：神经编解码器（Encodec、SoundStream、DAC）将音频压缩为 50-75 Hz 的离散词元，变压器或扩散模型生成词元。

**类型：** 建造
**语言：** Python
**先修：** 第 6·02 阶段（音频函数）、第 6·04 阶段（ASR）、第 8·06 阶段（DDPM）
**时间：** 约45分钟

## 问题

三个音频生成任务：

1. **文本转语音。** 给定文本，产生语音。干净的语音是窄带的，并且具有很强的语音结构，通过词元转换器很好地解决了这个问题。 VALL-E（微软）、NaturalSpeech 3、ElevenLabs、OpenAI TTS。
2. **音乐生成。** 根据提示（文本、旋律、和弦进行、流派），制作音乐。分布更广泛。 MusicGen（元）、稳定音频 2.5、Suno v4、Udio、Riffusion。
3. **音频效果/声音设计。** 根据提示，产生环境声音或拟音。 AudioGen、AudioLDM 2、稳定音频打开。

这三者都在相同的基础上运行：神经音频编解码器+ token-AR 或扩散生成器。

## 概念

![音频生成：编解码器词元+变压器或扩散](../assets/audio-generation.svg)

### 神经音频编解码器

编码解码器（Meta，2022）、SoundStream（Google，2021）、描述音频编解码器（DAC，2023）。卷积编码器将波形压缩为每个时间步向量；残差矢量量化 (RVQ) 将每个矢量转换为 K 个码本索引的级联。解码器将其反转。 2 kbps 下的 24 kHz 音频，使用 75 Hz 下的 8 个 RVQ 码本 = 600 tokens/sec.

```
waveform (16000 samples/sec)
    └─ encoder conv ─┐
                     ├─ RVQ layer 1 → indices at 75 Hz
                     ├─ RVQ layer 2 → indices at 75 Hz
                     ├─ ...
                     └─ RVQ layer 8
```

### 最上面的两种生成范式

**词元自回归。** 将 RVQ 词元压平为序列，运行仅解码器变压器。 MusicGen 使用“延迟并行”来与每个流偏移并行发出 K 个码本流。 VALL-E 根据文本提示 + 3 秒语音样本生成语音标记。

**潜在扩散。** 将编解码器标记打包为连续潜在变量或使用分类扩散对其进行建模。 Stable Audio 2.5 在连续音频潜伏上使用流匹配。 AudioLDM 2 使用文本到梅尔到音频的扩散。

2024-2026 年的趋势：流匹配在音乐方面获胜（更快的推理、更干净的样本），而 token-AR 仍然主导语音，因为它具有自然因果关系并且流式传输良好。

## 构建它

`code/main.py` 模拟了核心思想：在由两种不同“风格”生成的合成“音频标记”序列上训练一个微小的下一个标记转换器（风格 A 交替使用低和高标记，风格 B 使用单调斜坡）。款式和样品条件。

### 第 1 步：合成音频词元

```python
def make_tokens(style, length, vocab_size, rng):
    if style == 0:  # "speech-like": alternating
        return [i % vocab_size for i in range(length)]
    # "music-like": ramp
    return [(i * 3) % vocab_size for i in range(length)]
```

### 第 2 步：训练一个微型 token 预测器

以风格为条件的二元模型预测器。重点是模式：编解码器词元→交叉熵训练→自回归采样。

### 第 3 步：有条件采样

给定样式标记和起始标记，从预测分布中采样下一个标记。继续获得 20-40 个词元。

## 常见陷阱

- **编解码器质量限制了输出质量。** 如果编解码器不能忠实地表示声音，则生成器质量再多也无济于事。 DAC是目前开放最好的。
- **RVQ 误差累积。** 每个 RVQ 层都会对前一层的残差进行建模。第 1 层上的错误传播。在较高层上使用温度 0 进行采样会有所帮助。
- **音乐结构。** 30 秒的词元在 75 Hz 时为 20k+ 词元。对于变压器来说很难。 MusicGen采用滑动窗口+提示继续；稳定音频使用较短的剪辑+交叉淡入淡出。
- **边界处的伪影。** 生成的剪辑之间的交叉淡入淡出需要仔细的重叠添加。
- **清洁数据需求。** 音乐生成器需要数万小时的授权音乐。 Suno / Udio RIAA 诉讼（2024 年）使这一点浮出水面。
- **语音克隆伦理。** 3 秒的样本加上文本提示足以让 VALL-E / XTTS / ElevenLabs 克隆语音。每个生产模型都需要滥用检测+选择退出列表。

## 制作说明：音频是流媒体问题

音频是用户期望*在生成时*到达的一种输出形式，而不是一次性的。在生产方面，这意味着 TPOT（每个输出词元的时间）很重要，因为用户的收听速度是目标吞吐量，而不是他们的阅读速度。对于以 ~75 tokens/second（编码解码器）标记的 16kHz 音频，服务器必须为每个用户生成 ≥75 tokens/sec 才能保持播放流畅。

两个架构后果：

- **流匹配音频模型无法轻松进行流式传输。** Stable Audio 2.5 和 AudioCraft 2 一次渲染固定的剪辑长度。为了进行流式传输，您需要对剪辑进行分块并重叠边界（例如滑动窗口扩散），与编解码器 AR 模型相比，会增加 100-300 毫秒的延迟开销。

如果产品是“实时语音聊天”或“实时音乐延续”，请选择编解码器 AR 路径。如果是“提交时渲染 30 秒的剪辑”，则流匹配在质量和总延迟方面获胜。
