# 并行/群/网络架构
> 与主管对比：没有中央决策者。代理读取共享事件总线，异步获取工作，写回结果。 LangGraph 明确支持分散的动态环境的“群架构”。 Matrix (arXiv:2511.21686) 将控制流和数据流表示为通过分布式队列传递的序列化消息，以消除协调器瓶颈。权衡是明确的：确定性和可扩展性的可追溯性。 群体 适合具有许多独立子问题的任务；它不适合需要单一连贯计划的任务。
**类型：** 学习 + 构建
**语言：** Python（stdlib、`threading`、`queue`）
**先决条件：** 第 16 · 05 阶段（主管模式）、第 16 · 04 阶段（原始模型）
**时间：** ~75 分钟
＃＃ 问题
Supervisor 可以扩展到几个工人。几百个呢？主管本身就成为瓶颈：关于谁做什么的每一项决定都通过一个代理进行。一个缓慢的计划步骤会使整个系统陷入停滞。
群体架构颠覆了设计。工作人员不是由中央计划者调度工作，而是从共享队列中挑选工作。 “协调”已融入事件总线语义中。没有协调者；系统会扩展，直到队列扩展为止。
＃＃ 概念
### 形状
```
                ┌──── shared queue ────┐
                │                      │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     Worker  Worker  Worker   Worker
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            results pool
```

没有协调者。每个工作人员都会重复：拉取任务、处理、写入结果（以及可选的将后续任务排入队列）。
### 当群体适合时
- **许多独立的任务。** 抓取、转换、分类。任务不相互依赖。
- **可变持续时间工作。** 如果某些任务需要 100 毫秒，而其他任务则需要 10 秒，集群会自动平衡负载 - 快速工作人员会拉动下一个工作。主管必须预测持续时间。
- **吞吐量胜过确定性。** 您关心的是总完成时间，而不是严格的顺序。
### 当集群失败时
- **有序的工作流程。** 如果步骤 3 需要步骤 2 的输出，则集群可能会在步骤 2 完成之前触发步骤 3。
- **全局计划任务。** 复杂的研究问题受益于计划者。一群研究人员得出独立的事实，而不是连贯的报告。
- **调试。** 由于没有中央日志和异步工作，重现错误的成本很高。
### 矩阵 (arXiv:2511.21686)
Matrix 是 2025 年的论文，它让 群体 得出了自然的结论：控制流和数据流都是分布式队列上的序列化消息。没有中央协调员。容错能力来自于消息的持久性。可伸缩性是消息代理的问题，而不是系统的问题。
贡献：一种编程模型，其中多代理协调是“该代理订阅什么消息主题？”而不是“主管接下来选择哪个代理人？”这使得系统看起来像一个 pub/sub 事件网格。
### LangGraph 的 群体 架构
LangGraph 2025 文档明确将“群架构”描述为多代理模式之一：代理是节点，但边形成带有循环的有向图，并且可以从池中激活任何节点。工人根据条件而不是主管分配来选择可用的工作。
### 故障模式：饥饿和热点
如果所有工作人员都拉出最快的可用任务，那么长时间运行的任务永远不会被选中，直到它们成为唯一剩下的任务。经典的队列饥饿。
缓解措施：
- 具有显式老化的优先级队列（随着等待时间增加优先级）。
- 工人专业化：一些工人只承担“长”任务。
- 背压：限制进入队列的快速任务数量。
### 基于内容的路由链接
群体 自然地与基于内容的路由配对（第 22 课）。每种消息类型都有一个队列，而不是通用队列。专业工作者只订阅他们的类型。这是可扩展到数千个代理的消息总线架构的基础。
## 构建它
`code/main.py` 实现了从共享 `queue.Queue` 拉取的 4 个工作线程群。任务的持续时间各不相同（有些快，有些慢）。演示对比：
- **顺序基线：** 一名工作人员串行处理所有任务。
- **固定分配：** 每个任务预先分配给特定的工作人员（主管式）。
- **群体：**工作人员从共享队列中拉取。
群体自动平衡负载；当分配的任务较慢时，固定分配会使快速的工人闲置。
跑步：
```
python3 code/main.py
```

输出显示每个工作人员的任务计数（群体分布不均匀但最佳）和挂钟时间。
## 使用它
`outputs/skill-群体-fit.md` 评估任务是否应该使用 群体 与supervisor。输入：任务独立性、持续时间差异、订购要求、可调试性需求。
## 发货
清单：
- **具有老化的优先级队列。** 防止长时间任务饥饿。
- **工作线程幂等性。** 如果工作线程在运行中崩溃，则任务可能会被多次拉取。 Worker 必须是幂等的。
- **持久队列。** 使用 Kafka、Redis Streams 或数据库支持的队列进行生产。 `queue.Queue` 仅在内存中。
- **每个任务的可观察性。** 每个任务都有一个跟踪 ID；每个工作人员都用它记录 start/end。
- **背压。** 如果队列的增长速度快于工作人员耗尽队列的速度，则减慢生产者的速度。
## 练习
1.运行`code/main.py`。在可变持续时间工作负载上，群比顺序快多少？比固定分配快多少？
2. 添加优先级队列变体（使用 `queue.PriorityQueue`）。按任务“重要性”字段分配优先级。观察低优先级任务在连续负载下是否会出现饥饿现象。
3. 实现热点检测器：记录任何工作进程比最慢工作进程多 3 倍的任务。这对任务持续时间分布意味着什么？
4. 阅读 Matrix 论文 (arXiv:2511.21686) 摘要和第 3 节。确定 Matrix 接受的一项特定权衡（可扩展性增益）和它放弃的一项（可追溯性、确定性）。
5. 将 群体 演示转换为使用（task_type，payload）元组的 `queue.Queue`，工作人员仅订阅特定类型。当任务异构时，什么路由规则有意义？
## 关键术语
|术语 |人们怎么说|它实际上意味着什么 ||------|----------------|------------------------|
|群体架构| “去中心化代理” |工作人员从共享队列中拉出；没有中央协调器。 |
|活动巴士| “代理订阅主题”|消息代理按类型或内容将任务路由给工作人员。 |
|饥饿| “任务永远不会运行” |低优先级任务永远不会被选中，因为高优先级工作会不断到达。 |
|热点| “一名工人溺水”|负载不平衡，一名工人承担大部分任务。 |
|背压| “放慢制作人的速度”|当队列填满时向上游发出停止生产信号的机制。 |
|幂等工作者 | “安全重新运行” |处理两次的任务会产生相同的结果。必需的，因为工作人员可能会在运行中崩溃。 |
|耐用的队列| “在崩溃中幸存”|由磁盘或复制存储支持的队列；当工作线程崩溃时，任务不会丢失。 |
|矩阵框架| “完整的消息传递群”|数据流和控制流都是分布式队列上的序列化消息。 |
