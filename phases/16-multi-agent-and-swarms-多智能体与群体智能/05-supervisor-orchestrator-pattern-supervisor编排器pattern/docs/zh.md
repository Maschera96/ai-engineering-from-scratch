# 主管/协调员-工作人员模式
> 一名首席代理人计划和代表；专门的工作人员在并行环境中执行并报告。这是 Anthropic 研究系统背后的模式（Claude Opus 4 作为主导，Sonnet 4 作为子代理），在内部研究评估中测得比单代理 Opus 4 + 90.2%。 Anthropic 的工程帖子报告称，BrowseComp 上 80% 的差异仅由令牌使用来解释——多代理获胜很大程度上是因为每个子代理都获得了一个新的上下文窗口。本课程从原语构建主管模式，并涵盖生产部署中的 2026 年工程课程。
**类型：** 学习 + 构建
**语言：** Python（stdlib、`threading`）
**先决条件：** 第 16 阶段·04（原始模型）
**时间：** ~75 分钟
＃＃ 问题
研究是单代理系统失败的典型任务。您问“2023 年至 2026 年间多智能体系统发生了什么变化？”一个智能体依次阅读五篇论文，用文本填充一半的上下文，然后必须将所有这些论文放在一起进行推理。当它到达第五篇论文时，它会忘记第一篇论文。它无法并行化。
主管模式解决了这个问题：一个首席代理计划搜索，将每个子问题委托给一名工作人员，然后进行综合。每个工人对于一个狭窄的问题都有自己的 20 万代币窗口。领导永远看不到原始文件，只能看到工作人员的摘要。
Anthropic 的生产研究系统报告，与单个 Opus 4 相比，内部研究评估结果为 +90.2%。同一篇文章指出，80% 的 BrowseComp 差异是由“仅使用令牌”来解释的。每个子代理的新上下文是主要机制。
＃＃ 概念
### 模式
```
                 ┌──────────────┐
                 │   Lead       │  plans, decomposes,
                 │  (Opus 4)    │  synthesizes
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ Worker1 │  │ Worker2 │  │ Worker3 │
      │(Sonnet) │  │(Sonnet) │  │(Sonnet) │
      └─────────┘  └─────────┘  └─────────┘
         fresh       fresh        fresh
         context     context      context
```

领导从来不看原材料。在合成铅之前，工人们永远不会看到彼此的工作。每个箭头都是一个狭窄工件的切换。
### 为什么它会获胜
三种机制：
1. **每个子代理都有新鲜的背景。** 探索“FIPA-ACL 遗产”的工作人员不会携带领导计划花费的 40k 代币。对于一个问题，它有 200k 的窗口。
2. **通过提示进行专业化。** 领导的提示是“分解和综合”，而不是“研究”。每个工人的提示都很狭窄：“找出 X 中发生了什么变化。”有针对性的提示会产生有针对性的输出。
3. **并行性。** Worker 并发运行。挂钟时间大致为 `max(worker_times) + plan + synthesis`，而不是 `sum(worker_times)`。
### 工程课程 (Anthropic 2025)
Anthropic 帖子列出了几个仍与 2026 年相关的生产经验教训：
- **扩展查询的复杂性。**简单查询：一个代理，3-10 次工具调用。复杂查询：10 多个代理。领导必须对此进行估计，而不是呼叫者。
- **广泛然后狭窄。** 首先分解为广泛的子问题，然后如果答案需要深度，则为每个子问题产生更多的工作人员。
- **Rainbow 部署。** 代理是长期运行且有状态的。传统的蓝绿色不起作用。 Anthropic 使用彩虹：逐步推出新版本，同时旧版本耗尽。
- **令牌使用占主导地位。** 多代理的令牌约为单代理的 15 倍。仅当任务价值证明成本合理时才运行它。
### LangGraph 回合
LangGraph 最初附带了一个带有高级 `create_supervisor` 帮助程序的 `langgraph-supervisor` 库。 2025 年，LangChain 将建议改为直接通过工具调用来实现主管模式，因为工具调用可以更好地控制“主管看到的内容”（上下文工程）。图书馆仍然开放；文档现在推荐使用工具调用形式。
### 故障模式
- **领导对计划产生幻觉。** 如果领导生成的子问题无法分解真正的问题，那么工作人员就会对错误的目标进行精确研究。
- **工作人员过度探索。** 如果没有明确的范围界限，工作人员就会超出指定的子问题并污染综合步骤。
- **综合冲突。** 两名工人返回矛盾的事实。领导者必须重新询问（增加一轮）或明确指出分歧。默默地选择一方是最糟糕的失败：用户永远不知道发生了分歧。
### 当主管出错时
- **顺序任务。** 如果步骤 2 实际上需要步骤 1 的输出，则并行性没有任何意义。使用管道（CrewAI Sequential、LangGraph 线性图）。
- **简单的查询。** 单代理可以更快、更便宜地处理它们。在生成工人之前使用领导者的“规模努力”检查。
- **严格的决定论。** 主管使用法学硕士选择的委派。当 audit/replay 比适应性更重要时，静态图会更好。

## 构建它
`code/main.py` 使用 `threading` 实现三个并行工作线程的主管。领导者将查询分解为子问题，工作人员同时运行每个子问题，然后领导者进行综合。没有真正的法学硕士——工作人员被编写脚本来模拟获取和总结。
关键结构：
- `Lead.plan(query)` 将查询拆分为 3 个子问题。
- `Worker.run(sub_q)` 返回虚假摘要（可能是生产中任何使用工具的代理）。
- `Lead.run(query)` 启动线程、连接和合成中的工作程序。
跑步：
```
python3 code/main.py
```

输出显示计划、带有 start/end 时间戳的并行工作器跟踪以及最终综合。您可以看到挂钟获胜：三个 0.3 秒的工作进程运行时间约为 0.35 秒，而不是 0.9 秒。
## 使用它
`outputs/skill-supervisor-designer.md` 接受用户查询并生成主管模式设计：引导系统提示、工作人员角色、子问题分解规则和综合模板。在构建新的研究型代理系统之前使用它。
## 发货
部署主管模式之前的清单：
- **模型配对。** 领导推理层模型（Opus 类、`o3` 类）。工作人员使用更快、更便宜的模型（Sonnet、`o4-mini`）。
- **Worker 超时。** 任何超过 2× 中值运行时间的 Worker 都会被杀死；领先者要么以更窄的范围重新产生，要么在没有它的情况下继续前进。
- **每个工人的代币上限。** 硬性限制（比如预期综合输入的 10 倍）可以防止失控的工人超出预算。
- **可观察性。** 跟踪领导者的计划、每个工作人员的工具调用以及综合。这是任何事后调试的基础。
- **Rainbow 推出。** 有状态的长时间运行代理需要逐步版本转换，而不是热插拔。
## 练习
1. 运行 `code/main.py`，然后修改引导以生成 5 个工人，而不是 3 个。观察挂钟效果。在这个演示中，当工作线程数达到多少时，产生的开销会超过并行节省的开销？
2. 实现worker超时：杀死任何运行时间超过0.5秒的worker，并让lead综合剩余的结果。您需要什么可观察性才能知道工人被裁掉了？
3. 在领导者的综合中添加冲突检测步骤：如果两名工作人员返回相互矛盾的答案，则领导者会记录分歧，而不是选择一个。如何在不致电法学硕士的情况下发现矛盾？
4. 阅读 Anthropic 的研究系统工程帖子。列出这个玩具演示在生产中运行需要采用的三种做法。
5. 将 LangGraph 的 `create_supervisor`（旧版）与新的工具调用建议进行比较。这可以让您更好地控制主管所看到的内容？为什么 Anthropic 明确仅将子答案而不是将原始工作上下文传递到综合中？
## 关键术语
|术语 |人们怎么说|它实际上意味着什么 ||------|----------------|------------------------|
|主管| “首席代理人”|一个负责计划、委托和综合的协调代理。本身不做工作。 |
|工人| “子代理”|由主管调用的集中代理，具有狭窄的范围和自己的上下文窗口。 |
|协调员-工作者 | 《主管模式》 |一样的东西，不同的名字。 2026 文献同时使用了两者。 |
|新鲜的背景| “清洁窗户”|工作人员的上下文从其系统提示和分配的问题开始，而不是领导的历史记录。 |
|彩虹部署| “逐步推出” |长时间运行的有状态代理需要版本化的排出和替换，而不是蓝绿。 |
|代币主导地位 | “上下文是变量” |根据 Anthropic，80% 的研究评估方差来自使用的代币总数，而不是模型选择。 |
|规模化努力 | “将代理数量与复杂性相匹配” | Lead 估计查询难度，相应地产生 1 vs 10+ 工作人员。 |
|合成冲突| “工人们不同意” |两名工人返回相互矛盾的事实；领导者必须表达不同意见，而不是默默地挑衅。 |
