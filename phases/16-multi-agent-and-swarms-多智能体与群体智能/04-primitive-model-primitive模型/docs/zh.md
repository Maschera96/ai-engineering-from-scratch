# 多代理原始模型
> 2026 年发布的每个多代理框架（AutoGen、LangGraph、CrewAI、OpenAI Agents SDK、Microsoft Agent Framework）都是四维设计空间中的一个点。四个原语，仅此而已：代理、切换、共享状态、协调器。本课程从零开始构建它们，在所有四个上运行一个玩具系统，然后将每个主要框架映射到相同的轴上，以便您可以在一个段落中阅读任何新版本。
**类型：** 学习
**语言：** Python (stdlib)
**先决条件：** 第 14 阶段（代理工程）、第 16 阶段·01（为什么使用多代理）
**时间：** ~60 分钟
＃＃ 问题
每六个月就会发布一个新的多代理框架。 2023 年 AutoGen。2024 年 CrewAI。2024 年 LangGraph 和 OpenAI 群体。2025 年 4 月 Google ADK。2026 年 2 月 Microsoft Agent Framework RC。每个新闻稿都声称是“正确的抽象”。
如果你试图一次学习一个，你就会精疲力竭。 API 看起来不同。文档对于什么是“代理”存在分歧。一个框架将其共享内存称为“黑板”，另一个框架将其称为“消息池”，第三个框架将其称为“状态图”。你开始怀疑这个领域正在发生变化。
它不是。在营销的背后，四大原语是稳定的。学习一次，在一个段落中阅读每个新框架。
＃＃ 概念
### 四个原语
1. **代理** — 系统提示加上工具列表。无国籍；每次运行都从系统提示符和当前消息历史记录开始。
2. **切换** — 将控制权从一个代理转移到另一个代理的结构化转移。从机械上讲，这是一个返回新代理或符合条件的图形边缘的工具调用。
3. **共享状态** — 多个代理可以读取（有时可以写入）的任何数据结构。消息池、黑板、键值存储、向量内存。
4. **协调员** — 谁决定下一个发言者。选项：显式图（确定性）、LLM 发言者选择器（软）、最后发言者的切换呼叫 (OpenAI 群体) 或队列调度程序（群架构）。
这就是整个设计空间。每个框架都会为每个轴选择默认值；剩下的就是表面语法。
### 每个 2026 年框架如何与之对应
|框架|代理|交接 |共享状态|协调员||-----------|-------|---------|--------------|--------------|
| OpenAI 群体 / 代理 SDK | `Agent(instructions, tools)` |工具退货代理|来电者的问题 |法学硕士的下一次交接电话|
| AutoGen v0.4 / AG2 | `ConversableAgent` | GroupChat 上的发言者选择器 |消息池|选择器功能（LLM 或循环）|
|船员人工智能 | `Agent(role, goal, backstory)` | `Process.Sequential / Hierarchical` |任务输出链接 |经理LLM或静态订单|
| LangGraph |节点功能|图形边+条件| `StateGraph`减速器|图形，确定性 |
|微软代理框架|代理+编排模式|特定模式 |线程/上下文|特定模式 |
|谷歌 ADK |代理+A2A卡| A2A 任务 | A2A神器|主持人决定 |
表面差异看起来很大。下面：相同的四个旋钮。
### 为什么这很重要
一旦您看到原语，框架比较就变成了一个简短的清单：
- 编排器是否信任 LLM 进行路由（群体），或者是否在代码中固定路由（LangGraph）？
- 共享状态是完整历史记录（GroupChat）还是投影（StateGraph减速器）？
- 代理可以修改彼此的提示（CrewAI 管理器）还是只能交接（群体）？
这三个问题回答了 80% 的框架适合给定问题的问题。您不再购买“最好的多代理框架”，而是开始为您真正关心的轴进行设计。
### 无状态洞察
除了共享状态之外的每个原语都是无状态的。代理是（提示、工具）的功能。切换是一个函数调用。 Orchestrator 是一个调度程序。 **系统中唯一有状态的东西是共享状态。** 这就是所有有趣的错误所在：内存中毒（第 15 课）、消息排序、版本控制、写入争用。
隐藏共享状态的框架（群体）将问题推给调用者。集中它的框架（LangGraph 检查点、AutoGen 池）使其可检查，但将协调成本转移到共享状态实现上。
### 单个基元的剖析
＃＃＃＃ 代理人
```
Agent = (system_prompt, tools, model, optional_name)
```

没有记忆。无状态。具有相同系统提示和工具的两个代理可以互换。所有看起来像每个代理状态的东西实际上都处于共享状态或切换协议。
＃＃＃＃ 不可触摸
```
Handoff = (from_agent, to_agent, reason, payload)
```

三种实现占主导地位：
- **函数返回** — 工具返回下一个代理。这是 OpenAI 群体模式。代理在其工具模式中携带路由。
- **图边** — LangGraph。边缘是声明性的。 LLM 产生价值；条件选择下一个节点。
- **演讲者选择** — AutoGen GroupChat。选择器函数（有时本身就是 LLM 调用）读取池并选择下一个发言的人。
#### 共享状态
```
SharedState = { messages: [], artifacts: {}, context: {} }
```

至少是一个消息列表。通常更多：结构化工件（CrewAI 任务输出）、类型化上下文（LangGraph 减速器）、外部存储器（MCP、矢量 DB）。
两种拓扑：**全池**（每个代理看到每条消息）和**投影**（代理看到角色范围的视图）。完整的池很简单并且扩展性很差。预计池可以扩展，但需要预先进行架构设计。
#### 协调器
```
Orchestrator = ({state, last_speaker}) -> next_agent
```

四种口味：
- **静态** — 图表在构建时固定（LangGraph 确定性，CrewAI 顺序）。
- **LLM 选择** — LLM 读取池并选择下一个演讲者（AutoGen、CrewAI Hierarchical）。
- **切换驱动** — 当前代理通过调用切换工具（群体）来决定。
- **队列驱动** — 工作人员从共享队列中拉取；没有明确的下一个发言者（群体架构、矩阵）。
### 框架之间有什么变化
一旦基元被固定，剩下的设计决策是：
- **内存策略** — 短暂与持久检查点（LangGraph 检查点）。
- **安全边界** — 谁可以批准移交（人机交互）。
- **成本核算** — 每个代理代币预算。
- **可观察性** — 跟踪切换、保留重放状态。
所有这些都可以在原语之上实现。它们都不是新的原语。
## 构建它
`code/main.py` 在大约 150 行 stdlib Python 中实现了四个原语。没有真正的法学硕士——每个代理都是一个脚本化的策略，因此重点仍然放在协调结构上。
文件导出：
- `Agent` — 名称、系统提示、工具、策略函数的数据类。
- `Handoff` — 返回新代理的函数。
- `SharedState` — 线程安全消息池。
- `Orchestrator` — 三个变体：`StaticOrchestrator`、`HandoffOrchestrator`、`LLMSelectorOrchestrator`（模拟）。
该演示通过所有三种编排器类型运行相同的三代理管道（研究→写入→审查），并在最后打印消息池。您可以看到输出的不同之处仅在于*谁选择下一个*；代理和共享状态在运行中是相同的。
运行它：
```
python3 code/main.py
```

预期输出：三个协调器运行，每个模式一次。每个打印最终的消息池。如果研究人员决定尽早完成，那么由切换驱动的运行会到达更少的代理——这就是法学硕士路由权衡的缩影。
## 使用它
`outputs/skill-primitive-mapper.md` 是一种读取任何多代理代码库或框架文档并返回四基元映射的技能。在新的框架版本上运行它，以便在深入阅读文档之前获得一段理解。

## 练习
1. 使用不同的代理策略运行 `code/main.py` 3 次。观察编排器的选择如何改变运行的代理。
2. 实现第四种协调器类型：队列驱动的协调器类型，其中代理轮询工作的共享状态。可能会发生什么死锁？如何检测它？
3. 采用 LangGraph 快速入门 (https://docs.langchain.com/oss/python/langgraph/workflows-agents) 并将其重写为四个原语。 LangGraph 的哪些抽象映射 1:1，哪些是便利包装器？
4. 阅读 OpenAI 群体 食谱 (https://developers.openai.com/cookbook/examples/orchestrating_agents)。确定 群体 的四种原语中哪一种最符合人体工程学，以及将哪一种推送给调用者。
5. 在此表中找到一个完全隐藏共享状态的框架。解释当代理需要在不重新读取历史记录的情况下协调切换时会出现什么问题。
## 关键术语
|术语 |人们怎么说|它实际上意味着什么 ||------|----------------|------------------------|
|代理| “拥有工具的法学硕士”| `(system_prompt, tools, model)` 三元组。无国籍。 |
|交接 | “控制权转移” |命名下一个代理和可选负载的结构化调用。三种实现：函数返回、图边、说话者选择。 |
|共享状态| “记忆”/“语境” |多代理系统中唯一有状态的部分。消息池或黑板。 |
|协调员| “协调员” |谁决定谁下一个跑。静态图、LLM 选择器、切换驱动或队列驱动。 |
|原始| “抽象”|每个框架参数化的四个轴之一。不是框架功能。 |
|消息池| “共享聊天记录” |全历史共享状态。很容易推理，但扩展性很差。 |
|预计状态 | “范围视图” |共享状态的特定于角色的视图。规模化，需要架构设计。 |
|音箱选型| “下一个发言者是谁”| Orchestrator 模式，其中功能（通常是法学硕士）从组中选择下一个代理。 |
