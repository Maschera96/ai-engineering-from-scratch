# 生产扩展——队列、检查点、持久性
> 将多代理系统扩展到数千个并发运行需要**持久执行**。 LangGraph 的运行时在 `thread_id`（默认为 Postgres）键入的每个超级步骤之后写入一个检查点；工人崩溃释放租约，另一名工人恢复工作。代理可以无限期地休眠等待人类输入。 **MegaAgent** (arXiv:2408.09955) 运行每个代理的生产者-消费者队列，具有三种状态（空闲/处理/响应）和两层协调（组内聊天 + 组间管理员聊天）。 **Fiber/async** 击败了 LLM 流的每作业线程：线程在 99% 的时间里处于空闲状态等待令牌，光纤在 I/O. 上协同产出 对应点：Ashpreet Bedi 的“Scaling Agentic Software”主张 **FastAPI + Postgres + 没有其他**，直到负载证明情况并非如此 - 简单的架构比预期走得更远。本课程构建了一个持久的检查点日志、一个具有状态转换的每个代理工作队列、一个异步与线程演示，并实现了务实的“简单开始”规则。
**类型：** 学习 + 构建
**语言：** Python（stdlib、`asyncio`、`sqlite3`）
**先决条件：** 第 16 · 09 阶段（并行群网络）、第 16 · 13 阶段（共享内存）
**时间：** ~75 分钟
＃＃ 问题
原型多代理系统在一台笔记本电脑上运行，在内存事件循环中具有三个代理。您转向生产：
- 代理有时会运行数小时（长时间的研究、人机交互等待）。
- 工作进程崩溃。重新启动会丢失状态。
- 峰值负载是平均值的 10 倍；你需要水平缩放。
- 用户按代理运行付费；您需要一次性语义来进行充电。
内存中的事件循环不会执行这些操作。您需要在下面有一个持久的执行层。 2026 年的规范选项是：
1. 具有检查点的工作流引擎（Temporal、LangGraph 运行时）。
2. 具有状态存储的消息队列（Postgres + SQS/RabbitMQ）。
3. Actor-模型框架（MegaAgent 每个代理的生产者-消费者）。
4.手推FastAPI + Postgres（Bedi的论点）。
本课将构建每个模型的缩影。
＃＃ 概念
### 持久执行，模式
持久执行引擎在每个“步骤”（用 LangGraph 的语言来说是超级步骤）之后保留完整的程序状态。崩溃时：
```
worker crashes mid-step
  -> lease timeout
  -> another worker picks up the thread_id
  -> resumes from last checkpoint
  -> no duplicate side effects
```

其工作要求：
- **可序列化状态。** 所有代理状态都必须是可持久的。具有实时数据库连接的函数闭包无法生存。
- **确定性恢复。** 给定相同的状态和相同的输入，代理会产生相同的操作（或遵循 LLM 调用的外部确定性预言）。
- **幂等副作用。** 外部调用（工具调用、支付）必须是幂等的或使用重复数据删除密钥。
LangGraph 在每个超级步骤后写入一个检查点；每次活动后进行临时写入； Restate 使用事件来源的日志。所有三个都实现相同的模式。
### LangGraph 的运行时
每个代理都有一个`thread_id`； state 是一个类型化的字典；每个超级步骤都会向检查点表写入一行。恢复时，运行时从最后一个检查点开始重播，而不是从头开始。代理可以`interrupt()`等待人工输入；运行时持续存在并释放工作线程。当输入到达时，任何工作人员都可以恢复。
这是 2026 年 4 月的参考生产设计。
### MegaAgent 的每个代理队列
arXiv:2408.09955 描述了一项规模实验：一个集群中有数千个并发代理。建筑学：
```
agent i:
  state ∈ {Idle, Processing, Response}
  in_queue   <- messages addressed to agent i
  out_queue  -> replies + side effects

coordinators:
  intra-group chat  (agents in the same group)
  inter-group admin chat  (high-level routing)
```

两层协调使得组内对话密集发生，而组间对话保持稀疏——这种模式用于在数千个代理中保持成本线性。
### 异步与每个作业线程
LLM 调用是 I/O-bound. 等待下一个令牌的线程在 99% 的时间内处于空闲状态。每个线程消耗约 1MB RAM；如果有 10,000 个并发调用，则仅用于堆栈就需要 10GB。
Fibers（Python `asyncio`、Go goroutine、Rust `tokio`）在 I/O. 上协同产出同样的 10,000 个调用可以轻松处理。在 LLM-agent 规模上，异步并不是一种优化，而是一种架构。
例外：受 CPU 限制的后处理（嵌入、分词器技巧）仍然需要线程或进程。将 I/O 层与 CPU 层分开。
### 贝迪的对位法
“扩展代理软件”（Ashpreet Bedi，2026）认为大多数团队在测量负载之前就进行了过度设计。务实的默认设置：
- FastAPI + Postgres。
- 每个代理运行都是一行；通过乐观并发就地更新状态。
- 通过 `pg_notify` 或简单的 Celery 工作程序进行后台作业。
- 应用程序代码中的重试策略。
对于可管理任务上大约 100 个并发代理运行的负载，这通常就是您所需要的。当您测量失败时升级。
规则：当遇到简单架构无法解决的具体问题时，采用持久执行框架。过早的收养会浪费时间在仪式上，但不会带来回报。
### 恰好一次语义
对于付费代理运行，您需要“恰好一次有效”（至少一次交付+幂等消费者）。工程动作：
- **每次运行重复删除密钥。** 将其包含在每个副作用调用中。
- **发件箱模式。** 副作用首先写入表，然后由单独的进程执行它们。这两个步骤都是幂等的。
- **补偿事务。** 当副作用成功但其跟踪写入失败时，安排补偿。
这些是数据库工程模式，不是 LLM 特定的。 LLM税只是LLM打电话慢；其他一切都是标准的分布式系统。
### 彩虹部署
Anthropic 的多代理研究系统使用“彩虹部署”：代理运行时的多个版本同时运行，因此不必在每次代码部署时终止长时间运行的代理。金丝雀新版本的流量切片；当他们的代理完成时，淘汰旧版本。
这是长期运行的有状态系统的标准； 2026 年的调整是代理可以存活数小时，因此部署周期必须适应。
### 规范的生产清单
- 持久状态（检查点、快照或发件箱+可重播日志）。
- 幂等副作用。
- 用于 LLM 调用的异步 I/O 层。
- 通过重复数据删除至少一次传送。
- 用于有状态工作负载的 Rainbow/canary 部署。
- 可观察性：每个代理跟踪、超步审计、重试计数器。
## 构建它
`code/main.py` 实现：
- `CheckpointStore` — SQLite 支持的带有线程 ID 键的检查点日志。每个超级步骤都会附加一行。
- `run_with_checkpoint(agent, thread_id)` — 模拟运行中崩溃；第二个工作人员从上一个检查点恢复。
- `AgentQueue` — 每个代理空闲/处理/响应状态机，具有小型工作队列。
- `demo_async_vs_threads()` — 通过 asyncio 和线程运行 500 个并发模拟“LLM 调用”；报告挂钟和峰值内存（近似值）。
跑步：
```
python3 code/main.py
```

预期输出：模拟崩溃后检查点恢复成功；异步版本可在 1 秒内处理 500 个并发调用；线程版本需要几秒钟的时间，并且每个并发单元使用更多数量级的内存。

## 发货
规范生产强化：
- **从简单开始（Bedi 规则）。** FastAPI + Postgres 直到您测量它失败。
- **在优化之前检测一切。** 每次运行延迟直方图、每步时间、重试计数、故障分类。
- **副作用的发件箱模式。** 特别是支付和外部 API 调用。
- **Rainbow 部署。** 切勿在部署期间终止正在运行的代理运行。
- **当**遇到特定问题时采用持久执行引擎（Temporal / LangGraph / Restate）：长达一小时的人机循环等待、跨区域协调、复杂的 retry/compensation 策略。
- **I/O 层的异步。** 线程仅用于受 CPU 限制的后处理。
## 练习
1.运行`code/main.py`。确认检查点恢复工作；测量异步与线程并发的差异。
2. 实现一个**发件箱**表：每个工具调用首先写入发件箱，然后执行单独的 goroutine/task。通过运行工具调用两次来验证幂等性。
3. 模拟 **rainbow 部署**：两个并发运行时版本；将一半的新 thread_id 路由到每个线程；确认旧版本上正在运行的线程不会被中断。
4. 阅读 LangGraph 的运行时文档（链接如下）。确定在手工制作的 FastAPI + Postgres 版本中复制运行时的哪些功能需要最长的时间。这是采用的理由，还是可以推迟？
5. 阅读 MegaAgent (arXiv:2408.09955) 第 3 节。两层协调（组内 + 组间管理员聊天）是明确的。概述如何将其映射到具有两个队列系列的消息队列。
## 关键术语
|术语 |人们怎么说|它实际上意味着什么 ||------|----------------|------------------------|
|持久的执行力 | “保持程序状态” |引擎在每个超级步骤后写入状态；崩溃恢复是确定性的。 |
|超步| “交易边界”|检查点之间的工作单元。 LangGraph 术语。 |
|线程_id | “代理运行标识符”|绑定检查点和恢复逻辑的键。 |
|幂等性 | “可以安全重试” |重复一种副作用会产生与一次尝试相同的结果。 |
|发件箱图案| “解耦副作用” |将意图写入表；一个单独的执行者执行并标记为完成。 |
|至少一次交付 | “可能的重复”|消息队列语义；重复数据删除密钥使消费者一次性生效。 |
|彩虹部署| “重叠版本”|长时间运行的工作负载期间多个运行时版本并发。 |
|异步光纤| “合作产出”|用户模式并发；与 I/O-bound 加载的线程相比便宜。 |
|检查站| “状态快照” |超步边界处的串行化状态；简历的关键。 |
