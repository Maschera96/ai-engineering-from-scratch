# 失败模式：MAST、群体思维、单一文化、级联错误

> 2026 年的参考分类法是 **MAST**（Cemri 等，NeurIPS 2025，arXiv:2503.13657），它来自 7 个先进开源 MAS 的 1642 条执行轨迹，显示 **41 到 86.7% 的失败率**。三个根类别是：**规范问题**（41.77%），角色模糊、任务定义不清；**协调失败**（36.94%），通信中断、状态不同步；**验证缺口**（21.30%），缺少验证、缺少质量检查。**群体思维**家族（arXiv:2508.05687）补充了：单一文化崩塌（相同基础模型导致相关失败）、从众偏差（智能体互相强化错误）、心智理论不足、混合动机动态、级联可靠性失败。级联示例：重试风暴，支付失败触发订单重试，订单重试触发库存重试，库存服务被压垮（几秒内 10 倍负载，需要熔断器）。记忆投毒：一个智能体的幻觉进入共享记忆，下游智能体把它当成事实；准确率逐步衰减，让根因诊断变得痛苦。**STRATUS**（NeurIPS 2025）报告，通过专门的检测、诊断、验证智能体，缓解成功率提升 1.5 倍。本课把失败模式当成一等工程目标。

**Type:** Learn
**Languages:** Python (stdlib)
**Prerequisites:** Phase 16 · 13 (Shared Memory), Phase 16 · 14 (Consensus and BFT), Phase 16 · 15 (Voting and Debate Topology)
**Time:** ~75 minutes

## 问题

多智能体系统在真实任务上的失败率为 41 到 86.7%（Cemri 等 2025 在 7 个开源 MAS 上测得）。这不能靠“再加几个智能体”来调试。失败有结构性原因。MAST 分类法给出了类别。本课把每个类别映射到具体的检测、诊断和缓解模式，让这些数字不再像随机噪声。

2026 年的生产实践是把失败模式当成设计输入。你的架构还不能算“足够好”，除非你能指向每个 MAST 类别，并说出你部署了什么缓解手段。

## 概念

### MAST 类别

**规范问题（41.77% 的失败）。** 智能体的任务定义不够紧。示例：

- 角色模糊：两个智能体都以为自己是评审者。
- 任务描述不足：“总结这个”，但用户想要某个特定角度。
- 成功标准隐含：智能体无法判断自己是否成功。

缓解：
- 编写明确的角色契约。每个智能体的提示词说明它做什么，*以及它不做什么*。
- 每个任务都有验收测试。智能体开始前，定义“完成看起来像 X”。
- 启动前规范检查：由单独智能体在派发前审查任务定义。

**协调失败（36.94%）。** 通信或状态断裂。

示例：
- 两个智能体在没有同步的情况下更新共享状态。
- 智能体之间消息丢失（队列失败、超时）。
- 状态漂移：智能体 A 认为任务已完成；智能体 B 仍在执行。

缓解：
- 带版本的共享状态和乐观并发。
- 关键消息显式确认（重试直到 ack）。
- 定期状态同步检查点，尽早发现漂移。

**验证缺口（21.30%）。** 输出没有独立检查。

示例：
- 一个智能体声称成功；没人验证。
- 智能体链条中的每一环都信任前一环输出。
- 涌现的组合行为缺少测试覆盖。

缓解：
- 独立验证智能体（第 13 课）。只读，访问独立来源。
- 显式交接契约：“A 的输出必须先通过检查器 C，B 才能开始。”
- 结果日志，用于事后分析。

### 群体思维家族（arXiv:2508.05687）

当智能体同质化或互相模仿时，会出现五类相关失败：

**单一文化崩塌。** 相同基础模型或训练数据导致相关错误。三个智能体共享一个 LLM 时，也会共享它的幻觉。

**从众偏差。** 智能体向声音最大或最自信的同伴靠拢，即使对方是错的。

**ToM 不足。** 智能体无法建模彼此的信念；协调瓦解（第 18 课）。

**混合动机动态。** 激励部分对齐的智能体漂向折中中点，结果谁都不满意。

**级联可靠性失败。** 一个组件的错误模式触发依赖组件中的错误模式。

### 级联示例：重试风暴

一个经典的 2026 事故模式：

```
payment service fails 10% of requests
   ↓
order agent retries payment (exponential backoff but naive)
   ↓
each retry is a new order-inventory check
   ↓
inventory service sees 2x normal load
   ↓
inventory service starts timing out
   ↓
every order retries inventory check
   ↓
inventory service sees 10x normal load
   ↓
cluster goes down
```

修复方法很经典：**熔断器**。当下游错误率超过阈值时，用缓存结果或默认结果短路。再加上每个请求的重试预算上限。

熔断器是少数可以直接从分布式系统借用到多智能体失败缓解中的模式，不需要改造。

### 记忆投毒（回顾）

来自第 13 课：一个智能体的幻觉变成共享记忆事实；下游智能体基于被投毒的事实推理。用 MAST 术语说，这是共享记忆层的验证缺口。

症状是准确率逐步衰减。你不会得到崩溃，只会得到缓慢漂移，而且很难定位根因。

缓解：追加式日志、来源记录、不可写验证器。第 13 课已经覆盖。

### STRATUS：用于失败检测的专门智能体

STRATUS（NeurIPS 2025）报告，部署以下角色时，缓解成功率提升 1.5 倍：

- **检测智能体。** 监控症状模式（高分歧、重试尖峰、准确率漂移）。
- **诊断智能体。** 根据症状，从 MAST 分类法推断可能根因。
- **验证智能体。** 缓解措施应用后，检查症状是否消失。

这是应用到智能体系统上的 SRE 式事故响应。三个角色都可以是带专门提示词的 LLM 智能体。

### 失败模式审计

2026 年的最佳实践是每年（或每次大版本发布）做一次失败模式审计：

1. **轨迹样本。** 收集约 1000 条真实执行轨迹。
2. **分类。** 对每条轨迹中的失败，映射到 MAST + 群体思维类别。
3. **计算按类别的失败率。** 哪些类别主导你的系统？
4. **缓解排序。** 哪个修复能消除最多失败？
5. **选择 2 到 3 个缓解措施。** 实现；下个季度重新审计。

纪律比具体选择更重要。没有审计，失败会混进噪声里，永远不会被系统性处理。

### 当系统静默失败

最危险的失败类别是静默正确性失败。大声失败的系统（崩溃、异常、告警）可以被监控。产出貌似合理但错误结果的系统，无法通过异常日志发现。这就是为什么验证缺口虽然只占 21.30%，但按单次失败成本看最贵。

投入到：
- 基于样本的人工评审。
- 黄金数据集回归测试。
- 对重要输出做跨智能体交叉检查。

### 失败与慢性失败

有些失败立即发生，有些失败很慢。立即失败（超时、模式不匹配、认证错误）检测成本低。慢性失败（记忆投毒、单一文化漂移、角色模糊）检测和预防成本高。

2026 年的工程动作：为慢性失败代理指标做监控，这样你能在漂移变成可见错误前抓住它。共识率、重试率、输出长度分布、连续智能体版本之间的编辑距离，都是有用代理指标。

## 构建它

`code/main.py` 实现：

- `FailureTaxonomy`：把模拟事故分类到 MAST + 群体思维类别。
- `CircuitBreaker`：经典模式；错误率超过阈值时打开。
- `RetryStormSimulator`：展示级联失败；打开或关闭熔断器。
- `DetectionAgent`：脚本化的 STRATUS 风格症状匹配器。

运行：

```
python3 code/main.py
```

预期输出：
- 没有熔断器时的重试风暴：库存错误爆炸（模拟）。
- 有熔断器时：在阈值处封顶；提供降级模式响应。
- 检测智能体标记该模式并命名 MAST 类别。

## 使用它

`outputs/skill-mast-auditor.md` 对多智能体系统运行 MAST 风格失败模式审计。轨迹到分类，再到缓解排序。

## 交付它

生产中的失败模式纪律：

- **每季度做 MAST 审计。** 不是每年。类别会随着系统增长而变化。
- **到处放熔断器。** 每个到依赖服务的出站调用。默认打开阈值为 5 到 10% 错误率。
- **黄金数据集。** 小型、高质量、人工审计。每周做回归测试。
- **STRATUS 三件套。** 检测 + 诊断 + 验证智能体监控生产。先只从检测智能体开始；当症状有噪声时再加诊断。
- **失败预算。** 按类别给失败率设置显式 SLO。超出预算会触发停止发布讨论。

## 练习

1. 运行 `code/main.py`。确认熔断器限制了重试风暴。改变失败阈值并观察取舍。
2. 实现一个**慢性失败代理指标**：3 个并行智能体的共识率。当它急剧下降时触发告警。通过逐步让智能体输出相关，模拟单一文化漂移。
3. 阅读 Cemri 等（arXiv:2503.13657）。选择他们的 7 个 MAS 系统之一，映射其前 3 个失败类别。这些和 MAST 的预测相比如何？
4. 阅读 Groupthink 论文（arXiv:2508.05687）。指出五种模式中哪一种在生产中最难检测。提出一个代理指标。
5. 为你熟悉的某个多智能体系统设计 STRATUS 风格的检测、诊断、验证三件套。检测关注哪些症状？诊断建议哪些缓解措施？验证如何确认它们有效？

## 关键术语

| Term | What people say | What it actually means |
|------|----------------|------------------------|
| MAST | “2026 分类法” | Cemri 2025；3 个根类别 + 14 个失败子类型。 |
| Specification Problem | “角色模糊” | 任务或角色定义不足；智能体不知道该做什么。 |
| Coordination Failure | “状态漂移” | 智能体之间的通信或同步断裂。 |
| Verification Gap | “没人检查” | 输出没有独立验证就被接受。 |
| Groupthink family | “同质化失败” | 单一文化、从众、ToM 不足、混合动机、级联。 |
| Monoculture collapse | “同一模型，同一幻觉” | 来自共享基础模型或训练数据的相关错误。 |
| Retry storm | “级联错误放大” | 一个失败触发重试，重试放大下游负载。 |
| Circuit breaker | “按错误率快速失败” | 错误率超过阈值时打开；用默认结果短路。 |
| STRATUS | “事故响应三件套” | 检测 + 诊断 + 验证智能体。缓解成功率提升 1.5 倍。 |
| Memory poisoning | “幻觉传播” | 共享记忆事实被污染；下游智能体基于毒素推理。 |
