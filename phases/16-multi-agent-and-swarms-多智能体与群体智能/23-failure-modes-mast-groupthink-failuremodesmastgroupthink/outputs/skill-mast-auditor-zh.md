给定一个多智能体系统和抽样执行轨迹，运行失败模式审计。

产出：

1. **样本构建。** 至少 200 条生产轨迹，按任务类型和时间窗口均匀抽样。记录抽样方法和偏差风险。
2. **分类遍历。** 对每条轨迹标记 `success | failure`。对失败，分配一个 MAST 类别（spec / coord / verify），适用时再分配一个或多个群体思维家族标签（monoculture / conformity / tom / mixed-motive / cascade）。
3. **分布表。** 按 MAST 类别和群体思维标签统计计数和百分比。与 Cemri 2025 的参考分布（41.77 / 36.94 / 21.30）比较。与参考分布偏差很大的系统，通常有某个特定薄弱层。
4. **最高频失败模式。** 找出 3 个最常见的具体模式（例如，“两个智能体都在评审”）。记录复现步骤。
5. **缓解排序。** 对每个最高频模式，从标准库中提出缓解措施：显式角色契约、带版本的共享状态、独立验证器、熔断器、检测诊断验证（STRATUS）三件套。按该模式频率下的预期失败减少量排序。
6. **静默失败风险。** 有多少失败会产出貌似合理但错误的输出，而不是明显错误？静默率决定验证层投入。
7. **慢性失败代理指标。** 推荐 2 到 3 个线上指标，在漂移变成明显错误前暴露它：共识率、重试率、输出长度分布、智能体间编辑距离。

硬拒绝：

- 没有随机或分层样本的审计。手工挑选的失败会过度代表戏剧性案例，并漏掉慢性失败漂移。
- 没有基线测量的缓解建议。“加一个验证器”没有意义，除非已知当前失败率。
- 忽略 MAST-unknown 事故。如果一条轨迹不适合任何类别，说明分类法不完整；提出扩展，而不是强塞类别。
- 声称季度审计足够，却没有运行中的慢性失败监控。季度审计会漏掉审计之间的漂移。

拒绝规则：

- 如果轨迹缺少按智能体归因（谁写了什么，谁读了什么），审计无法区分协调失败和角色冲突。建议先添加结构化按智能体日志，再重新审计。
- 如果系统总失败轨迹少于 50 条，样本太小，无法生成分布估计。建议延长观察窗口。
- 如果轨迹包含 PII，先脱敏再分析。

输出：一份三页报告。用一句话摘要开头（“41% 规范失败，12% 协调失败，39% 验证缺口，8% unknown；最高频模式是双评审者冲突；ROI 最高的缓解是显式角色契约。”），然后给出上面的七个部分。最后给出优先行动列表：三个缓解措施，包含估计实现成本和预期失败率降低。
