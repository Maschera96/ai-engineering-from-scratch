# 分层架构及其失败模式
> 层次结构是主管嵌套的。经理代理高于下级经理高于工人。 CrewAI `Process.hierarchical` 是教科书版本：`manager_llm` 动态委托任务并验证输出。 LangGraph 等效项是 `create_supervisor(create_supervisor(...))`。当任务是真实的组织结构图时，这是自然的模式。这也是最有可能陷入管理循环的模式——经理代理分配工作不当、误解子输出或未能达成共识。顺序往往会打败它。
**类型：** 学习 + 构建
**语言：** Python (stdlib)
**先决条件：** 第 16 阶段·05（主管模式）
**时间：** ~60 分钟
＃＃ 问题
一旦主管模式生效，下一步自然就是“如果工人自己就是主管怎么办？”团队下设子团队；公司有部门又部门。分层架构反映了这一点。
问题：法学硕士经理与人类经理不同。人类管理者对其下属所了解的内容有稳定的先验。法学硕士经理每次都会根据其背景对组织进行重新推理。在这种情况下的微小偏差，整个树都会错误分配工作。
＃＃ 概念
### 形状
```
                 Manager
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       Sub-Mgr A         Sub-Mgr B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

每个内部节点都进行计划、委托和综合。只有叶子才起作用。
### 闪光之处
- **清晰的组织映射。** 如果真正的任务是部门级的（“法律审查文档，财务审查文档，工程审查文档，然后总结以供执行”），则层次结构是明确的。
- **本地汇总。** 每个下属经理都会在最高经理看到之前综合其团队的输出。最高管理者看到的是三个子管理者的摘要，而不是十五个工人的输出。
### 哪里破裂了
2026 年事后分析不断发现三种故障模式：
1. **任务分配错误。** 经理读取目标，幻想进行分解，然后委派给错误的下级经理。因为子管理者会乖乖地按照给定的要求工作，所以错误只会在顶层综合中浮现出来——与人类可能捕捉到的错误相比，少了一层。
2. **输出误解。** 子经理返回“无法验证声明 X”。高层经理总结为“X 说法未得到证实”。意义在各个层面上都有所变化。
3. **共识循环。** 两位副经理不同意；高层经理要求他们和解；他们重新下放权力；工人重新运行；子级经理返回的答案略有不同；环形。 CrewAI 的 `Process.hierarchical` 通过步数限制来防范这种情况，但限制本身现在是一个超参数。
### 决定性问题
顺序（线性管道）与分层：您的任务实际上是否有独立的子团队，或者它是一个伪装成一棵树的线性流程？如果是后者，请使用顺序。如果是前者，请使用分层但预算明确的调节规则。
### CrewAI 的实现
`Process.hierarchical` 将法学硕士经理与专业团队联系起来。经理：
- 接收顶级任务，
- 向工作人员分配子任务，
- 评估机组人员的产出，
- 决定是否接受、重新委托或迭代。
文档：https://docs.crewai.com/en/introduction（在核心概念下查找“分层流程”）。
### LangGraph 的实现
LangGraph 使用嵌套的 `create_supervisor` 调用。内部supervisor有自己的图；外部监督者将内部图视为不透明节点。这比 CrewAI 的调试更清晰（您可以单独单步执行每个图），但更难表达树的动态重塑。
参考：https://reference.langchain.com/python/langgraph-supervisor.
## 构建它
`code/main.py` 运行 3 级层次结构：
- 最高管理者：将任务分为“工程”和“法律”分支，
- 工程副经理：分为“前端”和“后端”工作人员，
- 法务副经理：一名工人。
演示将快乐路径（每个人都同意）与**扰乱路径**进行了对比，其中最高管理者的分解将“法律”错误地标记为“财务”，并观察错误级联——副经理乖乖地做财务工作，最高综合者报告财务调查结果，最初的法律问题没有得到解答。
跑步：
```
python3 code/main.py
```

输出显示两条路径，并清晰并排显示“询问的内容”与“交付的内容”。
## 使用它
`outputs/skill-hierarchy-fitness.md` 评估给定任务是否应使用分层、顺序或平面管理程序。输入：任务描述、组织结构、调节预算。输出：模式推荐以及要防范的特定故障模式。
## 发货
如果您发布分层：
- **帽树深度为 2。** 三个级别已经隐藏了大多数可观察性错误。
- **明确的调节预算。** 在最高管理者必须提交之前设置最大轮数。通常为 2。
- **每次合成的出处。** 每个节点的摘要必须引用哪个叶输出产生它。
- **分解漂移警报。** 记录管理者每一步的分解；与用户查询进行比较。如果分解不再涵盖查询，则发出警报。
## 练习
1. 运行 `code/main.py` 并比较高兴与不安。在最高层的输出完全偏离用户的问题之前，需要经过多少级的经理交接？
2. 添加第三级（顶层→子→子子→工人）。测量随着深度的增长，扰动路径自行纠正与完全发散的频率。
3. 在每个子经理处实施一个“金丝雀”工作人员，该工作人员始终询问原始用户问题不变。使用金丝雀答案来检测分解漂移。当金丝雀不同意综合答案时，经理应该如何反应？
4. 阅读 CrewAI 的 `Process.hierarchical` 文档。确定 CrewAI 应用的一种具体护栏（步骤限制、manager_llm 约束）并描述其针对的故障模式。
5. 将嵌套的 LangGraph 管理程序与 CrewAI 分层结构进行比较。这使得调节循环的检测成本更低？
## 关键术语
|术语 |人们怎么说|它实际上意味着什么 ||------|----------------|------------------------|
|分层 | “组织结构图模式” |监事之上的监事；只有叶子起作用。 |
|法学硕士 | 经理“老板”| LLM 在内部节点进行分解、分配和验证。 |
|分解漂移| “老大失去了阴谋”|高层管理者的分歧不再涵盖原来的问题。 |
|调节循环 | “无休止的会议”|分管领导不同意；高层重新代表；工人重新运行；循环直到预算用完。 |
|深度 2 天花板 | “不要深入超过 2 层”|经验护栏：3 个以上的级别会破坏可观察性。 |
|金丝雀问题| “各个层面的真实情况”|始终要求原始查询保持不变的工作人员，以检测漂移。 |
|来源链| “谁说了什么”|从每个合成追溯到产生它的叶输出。 |
