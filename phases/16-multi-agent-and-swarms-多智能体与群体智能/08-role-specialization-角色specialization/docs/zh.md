# 角色专业化——规划者、批评者、执行者、验证者
> 2026 年最常见的多智能体分解：一个智能体计划，一个智能体执行，一个智能体批评或验证。 MetaGPT (arXiv:2308.00352) 将其形式化为编码到角色提示中的 SOP——产品经理、架构师、项目经理、工程师、QA 工程师——遵循 `Code = SOP(Team)`。 ChatDev (arXiv:2307.07924) 通过“聊天链”与“沟通幻觉解除”（代理明确要求缺失的详细信息）连接设计师、程序员、审阅者、测试人员。验证器是承重的：Cemri 等人。 （MAST，arXiv：2503.13657）显示每个多代理故障都可以追溯到丢失或损坏的验证。普华永道 (PwC) 报告称，CrewAI 中的结构化验证循环的准确性提高了 7 倍 (10% → 70%)。
**类型：** 学习 + 构建
**语言：** Python (stdlib)
**先决条件：** 第 16 阶段·04（原始模型）、第 16 阶段·05（主管）
**时间：** ~60 分钟
＃＃ 问题
通用多代理系统产生通用输出。群聊中的三个程序员编写了相同平庸代码的三种风格。您可以添加更多代理，添加更多回合，但仍然不会超过质量阈值。
解决办法不是增加代理——而是“不同的”代理。分配不同的角色。为批评者提供规划者所没有的工具。为验证者提供客观的测试套件。现在系统内部对接地修正存在分歧，而不仅仅是平行猜测。
＃＃ 概念
### 四个典型角色
**规划者。** 读取目标，生成步骤列表或规范。工具：知识检索、文档。输出：结构化计划。
**执行者。** 一次读取一个计划步骤，生成工件。工具：实际工作工具（代码编译器、shell、API客户端）。输出：神器。
**批评家。** 根据计划者的意图读取执行者的输出。工具：工件的只读访问、静态分析。输出：accept/reject 以及原因。
**验证者。** 读取工件并运行确定性检查。工具：测试运行器、类型检查器、模式验证器。输出：带有证据的 pass/fail。
批评者是主观的、固执己见的，通常是法学硕士。验证者是客观的、确定性的、通常基于代码的。他们不是同一个角色。
### MetaGPT 的 SOP 模式
MetaGPT (arXiv:2308.00352) 将软件工程 SOP 编码为角色提示：
- **产品经理**撰写 PRD。
- **架构师** 进行系统设计。
- **项目经理** 分配任务。
- **工程师**实施。
- **QA 工程师** 运行测试。
每个角色都有严格的 input/output 架构。角色提示说明角色*是什么*以及它*必须产生什么*。 `Code = SOP(Team)` 公式 — 确定性 SOP 将法学硕士团队转变为可预测的管道。
### ChatDev 的沟通幻灭
ChatDev 添加了一个关键举措：当执行者需要计划中没有的特定细节时，它会在继续之前明确询问设计者。这可以防止法学硕士典型的合理发明细节的失败。
实现：角色提示包括“当您需要未提供的特定信息时，在生成输出之前按名称询问相关角色”。
### 为什么验证者最重要
塞姆里等人。 (MAST) 跟踪到 1642 个多代理执行失败。 21.3% 是验证差距——系统给出了一个没有人检查过的答案。剩下的 79% 通常可以追溯到“有一项检查悄无声息地失败或从未运行过”。验证是承载的角色。
普华永道报告（CrewAI 部署，2025 年）添加结构化验证循环将准确性从 10% 提高到 70%。一个角色可获得 7 倍收益。
### 批评者与验证者
- 评论家是审查工件质量的法学硕士。主观。可能会被看似合理的散文所愚弄。
- 验证程序是在工件上运行的确定性程序。客观的。为 pass/fail 提供证据。
两者都用。批评者发现了验证者无法阐明的品味问题。验证者捕获批评者看不到的错误，因为它们仅在运行时出现。
### 反模式
您系统中的每个角色都是法学硕士，每个角色的输出都是“对我来说看起来不错”。经典的 MAST 故障模式。添加至少一个验证者，其 pass/fail 由代码决定，而不是由 LLM 决定。
### 框架映射
- **CrewAI** — `Agent(role, goal, backstory)` 是教科书专业化表面。
- **LangGraph** — 节点可以有专门的提示；边缘强制管道。
- **AutoGen** — 群聊中具有单字名称的特定于角色的 ConversableAgent。
- **OpenAI Agents SDK** — 角色专用代理之间的切换工具。
## 构建它
`code/main.py` 实现了构建简单 Python 函数的 4 角色管道：
- **规划师**制定规格。
- **执行器**生成一个代码字符串。
- **评论家**（法学硕士模拟）标记明显的问题。
- **验证程序** 在沙箱 (`exec`) 中针对测试用例运行生成的代码。
演示运行两次：一次执行器生成正确的代码（批评者 + 验证者都通过），一次执行者生成不合规范的代码（批评者错过了错误，因为它看起来合理，验证者捕获它，因为测试失败）。
跑步：
```
python3 code/main.py
```

## 使用它
`outputs/skill-role-designer.md` 接受任务并生成角色名册（3-5 个角色）、每个角色的 input/output 架构以及验证者检查。在将代理连接到框架之前使用它。
## 发货
清单：
- **至少一名确定性验证者。** 绝不是全法学硕士。
- **每个角色显式 I/O 架构。** 规划器返回规范，而不是散文；执行器读取该模式。
- **沟通幻灭。** 当信息缺失时，执行者必须询问计划者；永远不要发明它。
- **Critic/verifier 排序。** 首先运行批评者（便宜，捕获设计问题），第二运行验证者（慢，捕获错误）。
- **循环预算。** 在升级为人类之前，最多进行 2 轮批评者-执行者修订。
## 练习
1. 运行 `code/main.py` 并观察验证者如何捕获批评者错过的错误。添加静态分析检查（计算 `return` 的出现次数）作为附加验证程序。运行时测试错过了什么？
2. 添加第五个角色：“需求分析师”，将用户愿望转化为可供规划人员使用的规范。应该向它提出什么样的沟通性解除幻觉请求？
3. 阅读 MetaGPT 第 3 节（“代理”）。列出 MetaGPT 5 个角色中每个角色的 input/output 架构。
4. 阅读 ChatDev 的聊天链图（arXiv:2307.07924 图 3）。确定沟通性幻觉解除在哪里打破了一个本来会无限的循环。
5. 普华永道 7 倍的准确度增益来自于验证循环。假设添加验证器无济于事的三个任务——确定性检查正确性是不可能的或昂贵的。
## 关键术语
|术语 |人们怎么说|它实际上意味着什么 ||------|----------------|------------------------|
|角色专业化| “不同的经纪人，不同的工作”|针对 planner/executor/critic/verifier 角色调整的独特系统提示。 |
| SOP模式| “编码标准操作程序”| MetaGPT 的框架：每个角色严格的 I/O 模式将团队变成管道。 |
|沟通性幻觉解除 | “发明之前先询问”| ChatDev 模式：执行者在细节缺失时询问计划者，而不是弥补细节。 |
|评论家 | “LLM审稿人”|主观、固执己见的审稿人。捕捉味道问题。可能会被看似合理的散文所愚弄。 |
|验证者 | “确定性检查” |基于代码的 pass/fail. 测试运行器、类型检查器、模式验证器。不能被愚弄。 |
|验证差距| “无人检查” | 21.3% 的 MAST 失败。答案在没有检查的情况下发送，这可能会发现该错误。 |
|修订循环 | “评论家将其退回”|批评家拒绝会触发执行程序重新运行并提供反馈。需要预算。 |
|全LLM反模式| “我觉得不错”|每个角色都是法学硕士，没有确定性检查。经典的 MAST 失败。 |
