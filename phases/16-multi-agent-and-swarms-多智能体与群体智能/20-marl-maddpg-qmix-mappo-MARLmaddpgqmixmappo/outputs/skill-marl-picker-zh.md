给定多智能体任务描述，选择 MARL 算法。
生产：
1. **任务分类。** 完全合作（共享奖励）、完全竞争（零和）、混合、普和。代理数量。同质与异质。
2. **可观察性。** 完全（每个代理都可以看到全局状态）、部分（每个代理只能看到自己的观察）或启用通信。
3. **动作空间。** 离散（类似 Atari，SMAC）或连续（粒子世界，MuJoCo）。影响算法选择。
4. **奖励结构。** 密集（每步形状）与稀疏（仅终端）。密集使得MAPPO实用；稀疏需要信用分配帮助（QMIX的值分解）。
5. **算法推荐。** 按照 Yu 等人的说法，以 MAPPO 作为基线开始。 2022.切换到：
   - QMIX 当需要合作+同质+强稀疏奖励信用分配时
   - MADDPG 混合时（合作+竞争）+连续行动
   - 单调性约束过于严格时的扩展（QTRAN、QPLEX、FACMAC）
6. **培训基础设施。** 您是否拥有：足够的交互数据、计算预算、奖励塑造专业知识、稳定性预算（每个实验 5-10 个种子）？如果没有，请为 LLM 代理推荐提示级策略。
7. **部署合约。** CTDE：在部署时，每个代理只能看到本地观察。明确地编写合约，以便运行时代码尊重它。
硬拒绝：
- 为第一次运行选择非 MAPPO 基线。 MAPPO 是 2026 年基线；从那里开始。
- 使用 QMIX 执行混合合作竞争任务。值分解假设单调聚合。
- 建议对缺乏交互数据或奖励信号的 LLM 代理系统进行 MARL 培训。在数据到位之前，即时级别的政策将表现得更好。
- 训练时不记录每个代理的观察结果和行动。调试是不可能的。
拒绝规则：
- 如果任务的交互数据少于 1000 集，建议提示级策略或监督微调。
- 如果任务是非马尔可夫的（需要记忆），但建议不包括反复出现的批评者，请标记差距。
- 如果任务是一般和竞争性的（多重均衡），MARL 不会单独选择一个；建议机制设计或平衡选择。
输出：一页的简介。从一句话建议开始（“具有集中价值函数的 MAPPO 基线；每个代理离散参与者；部署时的 CTDE；每个实验 5 个种子。”），然后是上面的七个部分。以培训到部署的流程结束：数据收集、培训、评估、推出。
