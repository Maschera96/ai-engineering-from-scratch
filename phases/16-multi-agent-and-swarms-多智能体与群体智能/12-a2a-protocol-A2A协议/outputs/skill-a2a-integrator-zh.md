给定两个需要互操作的代理系统，生成 A2A 集成计划：代理卡内容、任务模式、身份验证、传输模式。
生产：
1. **代理卡** 名称、版本、技能、端点、支持的模式（文本、结构化、图像、音频、视频）、协议版本、身份验证声明。
2. **每个技能的任务架构。** 输入 JSON 架构 + 工件 JSON 架构。明确——客户会验证。
3. **身份验证选择。** 承载令牌（OAuth2 或不透明）、mTLS 或签名请求。证明给定威胁模型（公共互联网、VPC、混合）的合理性。
4. **传输模式。** 轮询 vs SSE 流 vs webhook 回调。针对长时间运行或进度繁重的任务进行流式传输；轮询短期任务。
5. **速率限制。** 每个客户端和每个任务的限制。防止滥用。
6. **幂等性。** 重复 `POST /tasks` 请求的策略（客户端任务密钥、服务器端重复数据删除）。
7. **故障处理。** `failed` 之外的任务状态（可重试与致命）、死信策略、错误工件模式。
8. **MCP 与 A2A 拆分。** 如果远程代理在内部使用 MCP，请注意哪些工具是公开的，哪些工具保留在内部。
硬拒绝：
- 没有声明协议版本的代理卡。
- 当用例保证结构时，任务模式是自由格式的文本。
- 公共互联网部署上的 Auth=none。
拒绝规则：
- 如果两个代理在同一进程中运行，则拒绝 A2A 并建议直接调用 Python/JS。 A2A 用于跨系统边界。
- 如果延迟要求是往返 100 毫秒以下，则拒绝 A2A 并建议使用共享模式进行直接 RPC。
- 如果远程代理未声明代理卡，则拒绝集成并建议先发布一张。
输出：一页集成简介。关闭内联粘贴的代理卡 JSON，以便工程人员可以将其放入 `/.well-known/agent.json` 中。
