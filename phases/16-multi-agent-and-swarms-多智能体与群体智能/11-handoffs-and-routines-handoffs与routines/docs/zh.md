# 切换和例程 — 无状态编排
> OpenAI 的 群体（2024 年 10 月）将多代理编排提炼为两个原语：**例程**（指令 + 工具作为系统提示）和 **handoffs**（返回另一个代理的工具）。没有状态机，没有分支 DSL — LLM 通过调用正确的切换工具进行路由。 OpenAI Agents SDK（2025 年 3 月）是生产后续版本。 群体 本身仍然是最简洁的概念参考——它的整个源代码只有几百行。这种模式之所以如此流行，是因为 API 表面大致是“代理 = 提示 + 工具；切换 = 功能返回代理”。限制：无状态，因此内存是调用者的问题。
**类型：** 学习 + 构建
**语言：** Python (stdlib)
**先决条件：** 第 16 阶段·04（原始模型）
**时间：** ~60 分钟
＃＃ 问题
每个多代理框架都希望您学习其 DSL：LangGraph 节点和边缘、CrewAI 工作人员和任务、AutoGen GroupChat 和管理器。 DSL 是真正的抽象，但它们让事情感觉比实际需要的更重。
群体 朝相反的方向推进：使用模型已有的工具调用功能。切换变成了工具调用。协调者是当前进行对话的代理。状态机隐含在代理的系统提示中。
＃＃ 概念
### 两个基元
**例行程序。** 定义客服人员角色和可用工具的系统提示。可以将其视为一组有范围的指令：“您是分类代理；如果用户询问退款，请移交给退款代理。”
**Handoff。** 代理可以调用​​的工具，该工具返回新的代理对象。 群体 运行时检测 Agent 返回值并切换下一回合的活动代理。
这就是整个抽象。
```
def transfer_to_refunds():
    return refund_agent  # Swarm sees Agent return → switch active agent

triage_agent = Agent(
    name="triage",
    instructions="Route the user to the right specialist.",
    functions=[transfer_to_refunds, transfer_to_sales, transfer_to_support],
)
```

分类代理的系统提示使其根据用户消息选择正确的切换。 LLM 的工具调用负责路由。
### 为什么它如此病毒式传播
- **小 API。** 需要学习两个概念。
- **使用模型已有的功能。** 工具调用已经是跨提供商的生产级。
- **没有状态机负担。** 您无需描述图表；特工的提示描述了他们将工作交给谁。
### 无国籍贸易
群体 在运行之间是明确无状态的。该框架在运行期间保留消息历史记录，但它不会保留任何内容。内存、连续性、长时间运行的任务——所有调用者的问题。
在生产中（OpenAI Agents SDK，2025 年 3 月），这是发生的主要变化之一：SDK 添加了内置会话管理、护栏和跟踪，同时保留了切换原语。
### 当 群体/handoffs 适合时
- **分类模式。** 一线代理将用户路由给专家。
- **基于技能的交接。**“如果任务需要代码，请致电编码员；如果需要研究，请致电研究人员。”
- **简短、有限制的对话。** 客户支持、常见问题解答到工单、简单的工作流程。
### 当 群体 挣扎时
- **使用共享内存的长会话。** 切换会将对话状态重置为新代理的提示和历史记录。如果没有调用者管理的内存，代理之间就没有持久状态。
- **并行执行。** 切换是一次一次 — 活动代理进行切换。并行性要求调用者协调多个 群体 运行。
- **审核和重放。** 无状态运行很难准确重放；法学硕士的转学选择不是确定性的。
### OpenAI 代理 SDK（2025 年 3 月）
生产后继者添加了：
- **会话状态。** 跨运行的持久线程。
- **Guardrails.** Input/output 验证挂钩。
- **跟踪。** 每个工具调用和切换都会被记录。
- **切换过滤器。** 控制切换时传输的上下文。
切换原语仍然存在；围绕它添加了生产人体工程学。
### 群体 与 GroupChat
两者都使用 LLM 驱动的路由，但它们在**谁选择下一个**方面有所不同：
- GroupChat：选择器（功能或LLM）从外部选择下一个发言者。
- 群体：当前代理通过调用切换工具来选择其继任者。
群体 是“代理决定下一步”； GroupChat 是“经理决定下一步做什么”。 群体 的决策存在于主动代理的工具调用中； GroupChat 存在于 `GroupChatManager` 中。
## 构建它
`code/main.py` 从头开始​​实现 群体：一个代理数据类、一个切换机制（工具返回代理）和一个检测代理切换的运行循环。
演示：分类代理路由至退款、销售或支持专家。每个专家都有自己的工具。运行循环打印每次切换。
跑步：
```
python3 code/main.py
```

## 使用它
`outputs/skill-handoff-designer.md` 为给定任务设计切换拓扑：存在哪些代理、它们可以调用哪些切换、传输哪些上下文。
## 发货
清单：
- **切换日志记录。** 每次切换都会写入一个跟踪事件，其中包含从代理、到代理、上下文快照。
- **上下文传输规则。** 决定切换时移动的内容：完整历史记录（昂贵）、最后 N 条消息或摘要。
- **切换时的护栏。** 必须对具有不同工具权限的专家的切换进行身份验证 - 否则即时注入可能会强制进行不需要的切换。
- **循环检测。** 两个代理来回传递是一种常见的失败；通过简单的最后 K 环检查进行检测。
- **回退代理。** 如果切换目标不存在，则回退到安全默认值。
## 练习
1. 运行 `code/main.py`，分流至退款代理。确认第二轮活跃代理退款。
2. 添加环路检测规则：如果相同的两个座席连续交接3次，则强制退出。设计后备方案。
3. 阅读有关切换过滤器的 OpenAI Agents SDK 文档。实施“切换时总结”版本：传出代理在传入代理接管之前将上下文压缩为项目符号摘要。
4. 将 群体 切换与 GroupChatManager 选择器进行比较。哪种模式会使即时注入变得更糟，为什么？
5. 阅读 群体 食谱 (https://developers.openai.com/cookbook/examples/orchestrating_agents)。确定 群体 做出的一项明确的设计决策，即 OpenAI Agents SDK 已更改或保留。
## 关键术语
|术语 |人们怎么说|它实际上意味着什么 ||------|----------------|------------------------|
|日常 | “代理提示” |系统提示+工具列表。定义角色和可用的切换。 |
|交接 | “转移到另一个代理” |活动代理可以调用​​返回新代理的工具。运行时切换活动代理。 |
|无国籍| “运行之间没有记忆” | 群体 不会持久化任何东西；记忆是调用者的责任。 |
|活性剂| “现在谁在讲话” |目前正在进行对话的特工。切换改变了这一点。 |
|上下文传输 | “交接时会发生什么变化” |传入代理看到的历史记录的策略：完整、最后 N 或汇总。 |
|切换循环| 《乒乓特工》|两个代理不断互相交还的故障模式。 |
| OpenAI 代理 SDK | “生产群”| 2025年3月继任者；在切换原语之上添加会话、护栏、跟踪。 |
|切换过滤器| “转乘门” | SDK 功能可检查和修改切换边界处的上下文。 |
