# 共享内存和黑板图案
> 两种方法在 2026 年多代理系统中共存：**消息池**（每个人都可以看到每个人的消息，如 AutoGen GroupChat 或 MetaGPT 中）和 **带有订阅的黑板**（代理订阅相关事件，如上下文感知 MCP 或 Matrix 框架中）。两者都是多代理系统中唯一有状态的部分——这意味着两者都是有趣的错误所在。参考故障模式是**内存中毒**：一个代理幻觉一个“事实”，其他代理将其视为已验证的，并且准确性以一种比立即崩溃更难调试的方式逐渐衰减。本课程从 stdlib 构建这两种结构，注入中毒攻击，并展示在生产中实际起作用的三种缓解措施。
**类型：** 学习 + 构建
**语言：** Python（stdlib、`threading`）
**先决条件：** 第 16 · 04 阶段（原始模型）、第 16 · 09 阶段（并行群网络）
**时间：** ~75 分钟
＃＃ 问题
多代理系统需要代理共享事实的地方。字面上的选项是“传递消息中的所有内容”——但这通过额外的复制重新发明了共享状态。另一个是“给每个人一个全局日志”——但是全局日志无限增长并且很容易中毒。第三个是“为每个代理投射一个视图”——可扩展但架构繁重。
当其中一个代理产生幻觉并将幻觉写入共享状态时，读取该状态的每个下游代理都会将该幻觉视为事实。当人类注意到时，推理链已经有五步深，根本原因是有史以来写的第三条消息。调试多智能体准确性下降比调试崩溃更困难。
这就是记忆中毒。它是 MAST 分类中第二多记录的故障系列（Cemri 等人，arXiv:2503.13657），并且它是结构性的：任何没有出处和不可写验证器的共享内存设计最终都会展示它。
＃＃ 概念
### 两种主要拓扑
**完整的消息池。** 每个代理都会读取每条消息。 AutoGen GroupChat 和 MetaGPT 使用这个。简单、透明、可检查，但规模不会超过约 10 个代理，因为每个代理的上下文都充满了其他代理的工作。
```
agent-A ──write──▶ ┌────────────────┐ ◀──read── agent-D
                   │ message pool   │
agent-B ──write──▶ │                │ ◀──read── agent-E
                   │ (global log)   │
agent-C ──write──▶ └────────────────┘ ◀──read── agent-F
```

**黑板订阅。** 代理声明对主题感兴趣；底层仅路由相关消息。 CA-MCP (arXiv:2601.11595) 和 Matrix 去中心化框架 (arXiv:2511.21686) 使用它。可进一步扩展，但需要预先进行架构设计才能使订阅有意义。
```
                   ┌─ topic: prices ──┐
agent-A ──pub────▶ │                  │ ──▶ agent-D (subscribed)
                   ├─ topic: orders ──┤
agent-B ──pub────▶ │                  │ ──▶ agent-E (subscribed)
                   ├─ topic: alerts ──┤
agent-C ──pub────▶ │                  │ ──▶ agent-F (subscribed)
                   └──────────────────┘
```

### 当双方获胜时
- 当代理数量很少（< 10）、异构且对话时间较短时，**满池**获胜。当每个人都看到一切时，推理谁说了什么是微不足道的。
- 当代理数量众多、角色同质但实例数量众多（群）且对话持续时间较长时，**Blackboard** 会获胜。路由节省了令牌成本和上下文污染。
生产系统经常混合：顶部有一个小型完整池（计划层），下面有黑板（工人层）。
### 内存中毒，在一种情况下
三个特工正在执行一项研究任务。代理 A 是检索代理。特工 B 是一名总结者。特工 C 是一名分析师。
1. A 获取一个页面并向共享状态写入一条消息：“研究报告准确率提高了 42%。”
2. 获取的页面实际上显示“提高了 4.2%”。 A产生了小数的幻觉。
3. B 在阅读共享状态时写道：“据报道，准确率提高了 42%（来源：A）。”
4. C 在阅读共享状态时写道：“建议采用 — 42% 的提升具有变革性。”
5. 最终报告引用了从未存在过的 42% 的数字。
没有代理崩溃。没有测试失败。该系统“发挥了作用”。幻觉通过共享状态从一个代理的上下文跨越到每个下游代理的推理中。
### 为什么这是结构性的
如果没有共享状态，特工 A 的幻觉就会停留在 A 的上下文中。下游代理将重新获取或重新派生，并可能捕获错误。在幼稚的共享状态下，A 的上下文变成了每个人的上下文，幻觉被洗刷成事实。
问题不在于共享状态本身——它是共享状态**没有出处，也没有独立的验证者**。三个缓解措施解决了这个问题：
1. **每次写入的属性出处。** 共享状态中的每个条目都会记录谁编写的、何时、在什么提示下以及（如果适用）代理引用的来源。下游代理商对出处持怀疑态度。
2. **版本写入；将它们视为仅追加。** 更正是取代旧条目的新条目，而不是就地更新。审计跟踪被保留。
3. **保留至少一个无法写入共享状态的代理。** 只读验证器代理对条目进行采样、重新获取源并标记不一致。因为它无法写入池，所以不会受到池的毒害。
### 黑板先例（Hayes-Roth，1985）
黑板模式比法学硕士代理人早四十年。 Hayes-Roth（1985，“控制黑板架构”）描述了观察全局黑板、提供部分解决方案并触发其他来源的专业知识源。 2026 黑板（CA-MCP、矩阵）与 LLM 代理作为知识源和 JSON blob 作为部分解决方案的模式相同。旧文献记录了现代系统重新发现的写入争用、机会主义控制和一致性的解决方案。
### 投影与全视图
纯黑板为每个订阅者提供相同的投影（主题范围）。更积极的设计是**每个代理投影**：每个代理都会获得一个根据其角色定制的视图。 LangGraph 的状态缩减器是规范的 2026 实现 - 缩减器函数将全局状态折叠到特定于角色的切片中。
每个代理的投影可以进一步扩展，但需要一个模式。如果没有，您将在每个代理的提示中重建临时投影。
### 写竞争模式
多个代理同时写入是一个并发问题，而不仅仅是LLM问题。三种模式有效：
- **顺序写入器（单一生产者）。** 所有写入都通过一个串行化的协调器代理。简单，但是有瓶颈。
- **带有版本控制的乐观并发。** 每个条目都有一个版本；作者因版本不匹配而失败并重试。经典数据库技术。
- **主题分区。** 不同的代理拥有不同的主题。没有跨主题的争论。需要设计分区边界。
大多数 2026 框架默认使用顺序写入器，因为 LLM 调用足够慢，争用很少，而且瓶颈也不会造成影响。
### 不可写验证器
最能承受负载的缓解措施是只读验证器。实施细则：
- 验证者与团队共享状态（阅读黑板或池）。
- 验证者没有共享状态的写句柄——只有一个单独的验证通道。
- 验证者独立获取写入中引用的源。标记不同意见。
- 验证者自己的输出被路由到人类或单独的决策代理，永远不会反馈到池中。
如果没有这种分离，验证者的输出将成为池中的新条目，这意味着中毒池会毒害验证者，从而毒害其验证。
## 构建它
`code/main.py` 在 stdlib Python 中实现了两种拓扑以及玩具中毒攻击和三种缓解措施。
- `MessagePool` — 具有完整读出功能的线程安全仅附加日志。
- `Blackboard` — 具有每个代理订阅的主题键控 pub/sub。
- `ProvenanceEntry` — 每个写入记录（作者、时间戳、prompt_hash、source_uri）。
- `PoisoningScenario` — 运行三智能体研究任务，其中智能体 A 产生幻觉小数。打印最终报告。
- `Verifier` — 一个只读代理，用于重新获取源并标记不一致。在验证者在场的情况下运行相同的场景。
跑步：
```
python3 code/main.py
```

预期输出：
- 运行 1（无验证者）：幻觉 42% 传播到最终报告。
- 运行 2（使用验证者）：验证者标记不一致，池被标记为“已标记”，最终报告包括撤回。
## 使用它
`outputs/skill-memory-auditor.md` 是一项审核任何多代理系统的共享内存设计的技能，以了解出处、版本控制和验证者分离。在生产之前在新的多代理架构上运行它。
## 发货
对于任何共享内存设计：
- 记录每次写入的出处：`(writer, timestamp, prompt_hash, tool_calls_cited, source_uri)`。
- 使日志仅追加。更正是引用被取代条目的新条目。
- 部署至少一个具有独立源访问权限的只读验证器代理。
- 将验证器输出路由到单独的通道，而不是返回到共享池。
- 记录取代写入的比率——比率上升是幻觉模式的早期证据。
## 练习
1.运行`code/main.py`。确认运行 1 传播幻觉，运行 2 捕获幻觉。
2. 添加第二个幻觉：代理 B 发明了数据集大小。验证者应该捕获两者，而无需针对其中任何一个进行手动调整。
3. 将完整池切换到具有主题分区（`prices`、`summaries`、`analyses`）的黑板。主题分区会使哪些中毒场景变得更难实现，哪些场景没有帮助？
4. 阅读 Hayes-Roth（1985 年，“黑板控制架构”）。从本文中未讨论的论文中找出 2026 年系统将受益的两种控制模式。
5. 阅读 CA-MCP (arXiv:2601.11595)。将其共享上下文存储映射到 `code/main.py` 中的 MessagePool 或 Blackboard 类。 CA-MCP 在顶部添加了哪些原语？
## 关键术语
|术语 |人们怎么说|它实际上意味着什么 ||------|----------------|------------------------|
|消息池| “共享聊天记录” |每个代理都会读取的仅附加日志。完全透明，伸缩性差。 |
|黑板| “共享工作空间”|主题键 pub/sub. 代理订阅相关主题。规模更远。 |
|出处 | “谁写了什么” |每次写入的元数据：作者、时间戳、提示、来源。 |
|记忆中毒| “幻觉蔓延”|一个代理的错误进入共享状态，下游代理将其视为事实。 |
|仅附加| “没有就地更新” |更正是取代的新条目。保留审计跟踪。 |
|不可写验证器 | “独立审计师” |重新获取源并标记不一致的只读代理。 |
|投影| “范围视图” |根据全局状态计算的每个代理视图。 LangGraph 减速器是典型的情况。 |
|知识来源| “专家代理”| Hayes-Roth 1985 年对黑板参与者的称呼。 |
