给定多代理代码库或架构文档，审核共享内存设计并标记内存中毒暴露情况。
生产：
1. **拓扑。** 完整消息池、主题分区黑板、投影每个座席视图还是混合？命名数据结构（列表、字典、pandas 框架、向量存储、SQL 表）。统计稳定状态下作者和读者的粗略上限。
2. **出处字段。** 在每次写入时，条目是否记录：写入者 ID、时间戳、提示哈希或提示文本、工具调用跟踪、源 URI 或工具名称？列出存在的字段和缺失的字段。
3. **更新模型。** 日志是仅追加的，还是编写器会就地进行变异？如果发生突变，并发控制机制是什么（锁、乐观版本控制、无）？更正应该是取代条目，而不是就地编辑——标记任何不这样做的设计。
4. **验证者分离。** 是否有独立源访问的只读代理？它可以写入主池（不应该）吗？它的输出去哪里了？
5. **投影模式。** 如果设计使用投影（LangGraph 缩减器、黑板主题、角色范围视图），模式是否已记录？新代理如何声明他们消耗的投影？
6. **中毒风险评分。** 每个轴上评分 1-5：[来源完整性]、[对突变的取代]、[验证者独立性]、[投影模式清晰度]。在任何轴上得分低于 3 的系统将被标记。
硬拒绝：
- 任何未标记缺失验证者的审核。具有独立源访问的不可写验证器是负载缓解措施；没有它，所有其他缓解措施都是装饰性的。
- 建议“添加更多测试”的审核。测试不会发现内存中毒，因为中毒会产生通过测试的合理输出。
- 建议对内容进行哈希处理作为唯一来源的审核。哈希告诉您*写了什么*，而不是*谁*或*来自哪里*。
拒绝规则：
- 如果代码库在没有检查工具的情况下隐藏外部服务（Redis、Postgres、向量数据库）中的共享状态，请声明在没有生产读取访问权限的情况下审计无法完成。
- 如果系统的代理少于三个，请注意内存中毒风险较低，但出处仍然是便宜的保险。
- 如果系统使用具有内置状态管理的框架（LangGraph 检查点、AutoGen 池），请审核框架的保证而不是重新派生它们。
输出：一份两页的报告。从一句话总结开始（“共享状态是一个完整的消息池，没有出处，也没有验证者——中毒风险很高。”），然后是上面的六个部分。最后列出优先行动列表：三个更改，每个更改都标记为[关键]、[应该]或[最好有]，并附有预计的实施时间。
